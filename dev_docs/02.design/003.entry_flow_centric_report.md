# 진입 흐름 중심 리포트 설계

## 목적

- Legacy Java EE 분석기의 최우선 사용자 가치는 `어디로 들어와서 어떤 화면이 보이는가`를 빠르게 파악하는 것이다.
- 따라서 리포트의 1급 흐름 단위는 `URL/진입 패턴 -> Controller/Action -> View -> Layout`로 둔다.

## 핵심 원칙

- `Start Here`는 요약보다 `주요 진입 흐름`을 먼저 보여준다.
- `Entry Flow` 탭은 구조 나열이 아니라 실제 진입 흐름 후보를 보여준다.
- 각 진입 흐름 카드는 `요청 URL -> Dispatcher -> Controller -> View -> Layout` 체인을 한 줄로 먼저 보여준다.
- 관련 데이터 흐름이 있으면 같은 카드에서 `Service/DAO` 요약과 데이터 흐름 진입점을 함께 보여준다.
- `Data Flow`는 화면 진입 흐름의 보조 축으로 둔다.
- `Structure`, `Explore`, `Evidence`, `Raw`는 상세 확인용으로 뒤에 둔다.

## 데이터 우선순위

1. `web.xml`의 `url-pattern`과 `DispatcherServlet`
2. `contextConfigLocation` 기반 Spring 설정 힌트
3. `Controller/Action`의 `requestMappings`
4. `controller -> view` 관계
5. `view -> layout` 관계
6. 부족할 경우 `view-only`, `controller-only` fallback 카드

## 현재 구현 방향

- `requestMappings`를 `controller/action` 메타데이터에 저장한다.
- `web.xml`의 `url-pattern`, `servletName`, `contextConfigLocation`을 route/entry metadata에 저장한다.
- `Primary Entry Flows`와 `Entry Flow`는 위 신호를 합성한 명시적 흐름 카드를 사용한다.
- 화면 연결이 불완전해도 진입 URL이나 진입 패턴이 있으면 흐름 후보를 먼저 보여준다.
