# `Start Here` / `Flow Details` 우선 노출 기준

## 목적

- 현재 리포트가 `유지보수/온보딩 담당자` 기준으로 어떤 정보를 먼저 보여야 하는지 고정한다.
- `Flow Details`가 세부 정보 집합이 아니라 실제 추적 순서를 제공하도록 기준을 문서화한다.

## `Start Here` 우선 기준

### 1. 가장 먼저 보여야 하는 것

- `Framework Flow`
  - 왜 이 앱이 특정 URL 패턴을 받을 수 있는지 설명 가능해야 한다.
- 대표 `Screen Flow`
  - 신규 참여자가 가장 빨리 따라갈 수 있는 화면 요청 1~5개를 우선 노출한다.
- 대표 `Non-screen Flow`
  - 운영 장애나 연동 확인에 자주 쓰이는 API/다운로드 흐름 1~5개를 우선 노출한다.

### 2. 대표 흐름 선정 규칙

- concrete route가 wildcard보다 우선
- handler method가 식별된 흐름이 controller-only fallback보다 우선
- direct evidence가 있는 business/data path가 fallback보다 우선
- `heuristic` 기반 카드만 있는 경우 대표 흐름으로 올리지 않음
- 동일 controller에서 변형만 다른 흐름은 variant 수로 묶고 대표 route 하나를 먼저 노출

### 3. persona별 우선 읽기 경로

- 온보딩:
  - `Framework Flow -> 대표 Screen Flow -> Flow Details -> UI Actions`
- 아키텍처 리뷰:
  - `Framework Flow -> 대표 Screen/Non-screen Flow -> Architecture Context`
- 운영이관:
  - `대표 Non-screen Flow -> response kind / redirect / file hints -> Related Configs`

## `Flow Details` 고정 순서

- `1. Entry Setup`
- `2. Request Path`
- `3. Business Steps`
- `4. Data Access`
- `5. View/Response`
- `6. UI Actions`
- `7. Related Configs`

## 각 섹션의 역할

- `Entry Setup`
  - entry pattern, dispatcher, spring config, mapping file을 먼저 보여준다.
- `Request Path`
  - handler mapping, method resolver, bean, controller/action, handler method를 노출한다.
- `Business Steps`
  - controller -> service -> biz -> dao 체인과 evidence를 보여준다.
- `Data Access`
  - visible data path만 우선 노출하고 weak inferred는 기본 숨김 상태를 유지한다.
- `View/Response`
  - screen은 logical view / resolved JSP / layout, non-screen은 response kind / content type / redirect / file hints를 보여준다.
- `UI Actions`
  - 현재 화면에서 다음 요청으로 이어지는 링크를 배치한다.
- `Related Configs`
  - dispatcher config와 entry pattern 같은 근거 파일을 마지막에 둔다.

## 회귀에서 반드시 지켜야 하는 조건

- `Start Here`의 대표 flow 목록에 `heuristic` only 카드가 먼저 오면 실패
- `Flow Details`의 section key 순서가 바뀌면 실패
- `UI Actions`가 있는 화면 flow에서 해당 섹션이 빠지면 실패
- non-screen flow에서 `response kind`가 `View/Response` 섹션 밖으로 밀리면 실패
