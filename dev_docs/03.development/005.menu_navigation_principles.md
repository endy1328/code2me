# Menu And Navigation Principles

## 목적

- 문서, CLI 안내, 리포트 메뉴가 같은 사용자 흐름을 따르도록 상위 정보 구조 원칙을 고정한다.
- `Legacy Java EE` 단일 프로파일에 맞춰진 용어를 다음 프로파일 확장에도 버틸 수 있는 중립 구조로 정리한다.

## 핵심 원칙

### 1. 상위 메뉴는 사용자 흐름 중심

- 먼저 `무엇을 하는 도구인가`
- 그다음 `어떻게 실행하는가`
- 그다음 `리포트를 어떻게 읽는가`
- 이후에 `검증 상태`, `다음 프로파일`, `실행 트래커`로 내려간다

### 2. 상위 메뉴는 프로파일 중립 용어 사용

- 상위 메뉴에서는 특정 프레임워크 구현체 이름을 피한다
- `DispatcherServlet`, `StrutsFilter`, `ActionBean` 같은 표현은 세부 설명이나 카드 내부에서만 쓴다
- `Framework Flow` 대신 `Entry Flows`처럼 중립 표현을 우선한다

### 3. 읽기 순서는 유지

- `Start Here -> Entry Flows -> Request Flows -> Flow Details -> Context -> Evidence`
- 이 순서는 유지보수/온보딩 담당자와 아키텍처 리뷰어 모두에게 유효하다

### 4. 핵심 보기와 심화 보기를 분리

- 핵심 보기:
  - `Start Here`
  - `Entry Flows`
  - `Request Flows`
  - `Flow Details`
- 심화 보기:
  - `Context`
  - `Explore`
  - `Evidence`
  - `Raw / JSON`

### 5. README는 얇게, 운영 문서는 인덱스로 연결

- README는 개요, 실행, 리포트 읽기, 문서 인덱스만 맡는다
- 완료 기준, 회귀 매트릭스, 실제 프로젝트 게이트, 실행 트래커는 별도 문서로 링크한다

## 권장 상위 구조

### 문서

- `Product Status`
- `How To Use`
- `Report Navigation`
- `Docs Index`
- `Validation`
- `Next Profile`

### 리포트

- `Start Here`
- `Entry Flows`
- `Request Flows`
- `Flow Details`
- `Context`
- `Explore`
- `Evidence`
- `Raw / JSON`

## 현재 적용 원칙

- `Start Here`와 `Flow Details` 구조는 유지한다
- `Framework Flow`는 상위 라벨에서 단계적으로 `Entry Flows`로 옮긴다
- `Screen Flows`와 `Non-screen Flows`는 요청 흐름의 하위 분류로 설명한다
- `Architecture Context`는 지원 정보라는 성격이 드러나도록 `Context` 축으로 다룬다
