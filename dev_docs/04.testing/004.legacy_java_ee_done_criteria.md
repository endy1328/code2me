# Legacy Java EE 완료 기준

## 목적

- `Legacy Java EE` 분석이 “샘플에서 돌아간다” 수준을 넘어, 실제 레거시 프로젝트에서도 설명 가능한 흐름을 안정적으로 복원하는 상태를 완료 기준으로 고정한다.
- 이후 범용 분석기로 확장할 때도 재사용 가능한 `분석 코어`, `confidence`, `merge`, `report` 기준을 먼저 확보한다.

## 범위

- 현재 프로파일: `legacy-java-ee`
- 현재 목표: `web.xml -> Spring XML -> Controller/Action -> Service/Biz -> DAO -> Mapper/SQL -> View/Response` 흐름 복원
- 결과 산출물: `snapshot.json`, `summary.md`, `report.html`, `explore.html`, `evidence.html`, `raw.html`

## Persona

- 1순위 사용자: `유지보수/온보딩 담당자`
- 2순위 사용자: `아키텍처 리뷰어`
- 현재 단계에서는 운영이관/배포 정보보다 `어디로 들어와서 어떤 코드와 화면으로 이어지는가`가 우선이다.

## 필수 성공 기준

### 1. Entry Detection

- `web.xml`에서 servlet, `DispatcherServlet`, `url-pattern`, `contextConfigLocation`을 식별할 수 있어야 한다.
- 다수의 entry가 있을 때도 최소 1개 이상의 대표 진입 패턴이 리포트에 노출되어야 한다.
- `Framework Flow`와 `Start Here`에서 진입점이 설명 가능한 형태로 보여야 한다.

### 2. Request Mapping

- 아래 입력 계열 중 현재 fixture 범위에 포함된 규칙은 모두 복원 가능해야 한다.
  - annotation 기반 `@RequestMapping`
  - `SimpleUrlHandlerMapping`
  - `BeanNameUrlHandlerMapping`
  - `PropertiesMethodNameResolver`
  - XML multi-action controller
- request mapping은 최소 `controller`, 가능하면 `handler method` 단위로 리포트에서 검색 가능해야 한다.

### 3. Flow Reconstruction

- 화면 흐름은 최소 `URL -> Controller/Action -> View -> Layout`까지 복원되어야 한다.
- 비화면 흐름은 최소 `URL -> Controller/Action -> Response kind`까지 복원되어야 한다.
- service/biz/dao 연결이 있을 경우 `Flow Details`에 business path로 보여야 한다.
- controller-only, view-only fallback은 유지하되, 직접 근거가 있는 흐름보다 우선되면 안 된다.

### 4. View Resolution

- JSP view와 layout 관계를 식별할 수 있어야 한다.
- `SiteMesh` direct, alias, pattern 계열이 fixture 기준으로 안정적으로 복원되어야 한다.
- view resolver prefix/suffix가 있는 경우 logical view name과 resolved JSP 후보가 함께 보여야 한다.

### 5. Non-screen Classification

- non-screen flow는 최소 아래 종류로 분류되어야 한다.
  - `json`
  - `file`
  - `redirect`
  - `action`
  - `unknown`
- `response kind`, `content types`, `redirect target`, `file response hints`가 있을 경우 `Flow Details`에 함께 노출되어야 한다.

### 6. Data Flow

- `controller -> service -> dao` 또는 `controller -> service -> biz -> dao` 연결이 식별 가능해야 한다.
- `dao -> mapper -> sql` 직접 근거가 있으면 `confirmed`로 표시되어야 한다.
- 이름 기반 fallback만 있는 경우 `heuristic`으로 격하되어야 하며 기본 숨김 상태여야 한다.
- 외부 HTTP/API 호출은 `integration`으로 보존되어야 한다.

### 7. Confidence / Inference Policy

- `confirmed`, `inferred`, `heuristic`의 의미가 문서와 UI에서 일치해야 한다.
- `heuristic`, `low confidence`, `근거 종류 2개 미만` 카드는 기본 숨김 상태여야 한다.
- weak inferred path는 supporting card로만 노출되어야 하며, main flow를 대체하면 안 된다.

### 8. Report Behavior

- `Start Here`는 진입 흐름 우선 구조를 유지해야 한다.
- `Flow Details`는 `Entry Setup -> Request Path -> Business Steps -> Data Access -> View/Response -> UI Actions -> Related Configs` 흐름을 유지해야 한다.
- route 정렬은 wildcard보다 concrete route를 우선해야 한다.
- 대형 스냅샷에서도 `report.html`은 요약 중심, `explore/evidence/raw`는 분리 페이지 구조를 유지해야 한다.

## Merge / Conflict Policy

- 동일 node가 여러 adapter에서 관찰되면, 더 구체적인 path와 더 강한 confidence를 우선한다.
- request handler 정보는 합쳐지되, `requestMappings`, `viewNames`, `produces`, `contentTypes`, `redirectTargets`, `fileResponseHints`, `serviceCalls`는 모두 유실 없이 병합되어야 한다.
- annotation 기반 직접 근거가 XML fallback보다 우선한다.
- 직접 `queries` edge와 SQL call 근거가 있으면 fallback mapper match보다 우선한다.
- fallback은 기존 고신뢰 흐름을 덮어쓰지 않고 보조 근거로만 남아야 한다.

## 비완료 상태로 보는 조건

- direct evidence가 있는 흐름보다 fallback이 앞서 보이는 경우
- request handler 메타데이터가 merge 중 유실되는 경우
- `response kind` 또는 `data flow level`이 문서와 다르게 표시되는 경우
- 합성 fixture는 통과하지만 공개 샘플에서 반복적인 false positive가 다수 발생하는 경우
