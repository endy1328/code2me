# Action-family Legacy Web 완료 기준

## 목적

- `action-family-legacy-web` 분석이 Struts/Stripes 계열 레거시 웹앱에서 `entry -> action handler -> view/response` 흐름을 사람 기준으로 설명 가능한 상태를 완료 기준으로 고정한다.
- `legacy-java-ee`와 경계를 섞지 않고, action-family 전용 mapping / result / event 해석 범위를 별도로 관리한다.

## 범위

- 현재 프로파일: `action-family-legacy-web`
- 현재 목표: `web.xml filter -> action mapping -> handler method/event -> service -> JSP/result/response` 흐름 복원
- 결과 산출물: `snapshot.json`, `summary.md`, `report.html`, `explore.html`, `evidence.html`, `raw.html`

## Persona

- 1순위 사용자: `유지보수/온보딩 담당자`
- 2순위 사용자: `아키텍처 리뷰어`
- 현재 단계에서는 action URL이 어떤 핸들러와 화면으로 이어지는지가 우선이다.

## 필수 성공 기준

### 1. Entry Detection

- `web.xml`의 `StrutsPrepareAndExecuteFilter`, `StripesFilter`, `DynamicMappingFilter` 중 fixture 범위에 포함된 신호를 식별할 수 있어야 한다.
- 최소 1개 이상의 대표 entry pattern이 `Start Here`와 `Entry Flows`에 노출되어야 한다.
- `legacy-java-ee`와 경계가 섞이지 않고 `action-family-legacy-web`가 우선 선택되어야 한다.

### 2. Action Mapping

- 아래 입력 계열 중 현재 fixture 범위에 포함된 규칙은 모두 복원 가능해야 한다.
  - Struts 2 `struts.xml` action / method / result
  - Stripes `@UrlBinding`
  - Stripes `@DefaultHandler`
  - Stripes `@HandlesEvent`
- request mapping은 최소 `action class`, 가능하면 `handler method/event` 단위로 리포트에서 검색 가능해야 한다.

### 3. Flow Reconstruction

- 화면 흐름은 최소 `URL -> Action handler -> View -> Layout`까지 복원되어야 한다.
- 비화면 흐름은 최소 `URL -> Action handler -> Response kind`까지 복원되어야 한다.
- service 호출이 있을 경우 `Flow Details`에 business path로 보여야 한다.
- controller-only fallback은 유지하되, 직접 근거가 있는 action/view 흐름보다 우선되면 안 된다.

### 4. View / Result Resolution

- Struts result의 `dispatcher`, `redirect`, `redirectAction`, `stream` 중 fixture 범위에 포함된 종류는 식별 가능해야 한다.
- Stripes `ForwardResolution`, `RedirectResolution`, `StreamingResolution` 최소 해석이 유지되어야 한다.
- JSP와 action 간 `view-to-action reverse linkage`가 fixture 기준으로 안정적으로 복원되어야 한다.

### 5. Event / Response Classification

- event query 또는 submit name으로 분기되는 Stripes handler가 최소 fixture 기준으로 구분되어야 한다.
- non-screen flow는 최소 아래 종류로 분류되어야 한다.
  - `json`
  - `file`
  - `redirect`
  - `action`
  - `unknown`
- rendered JSP가 직접 연결된 flow는 가능하면 screen flow로 승격되어야 한다.

### 6. Confidence / Inference Policy

- `confirmed`, `inferred`, `heuristic`의 의미가 문서와 UI에서 일치해야 한다.
- weak inferred path는 supporting card로만 노출되어야 하며, main flow를 대체하면 안 된다.
- action route 추정은 direct `@UrlBinding` / `struts.xml` 근거보다 우선되면 안 된다.

### 7. Report Behavior

- `Start Here`는 대표 entry와 대표 action flow를 요약 중심으로 보여야 한다.
- `Request Flows`는 `screen` / `non-screen` 분리가 실제 결과와 어긋나지 않아야 한다.
- `Flow Details`는 `Entry Setup -> Request Path -> Business Steps -> Data Access -> View/Response -> UI Actions -> Related Configs` 흐름을 유지해야 한다.

## 비완료 상태로 보는 조건

- rendered JSP가 있는데도 대표 flow가 계속 non-screen으로만 남는 경우
- direct action/result 근거보다 package/name fallback이 앞서 보이는 경우
- event route와 default handler가 뒤섞이는 경우
- 합성 fixture는 통과하지만 공개 샘플에서 screen/api 분류가 반복적으로 어긋나는 경우
