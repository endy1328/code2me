## 현재 강한 영역

- annotation 기반 request mapping
- 다중 dispatcher별 entry context 선택
- `SimpleUrlHandlerMapping` + method resolver 기반 XML multi-action
- `BeanNameUrlHandlerMapping` 기반 XML-heavy controller
- direct SQL evidence 우선 / fallback 억제
- `SiteMesh` direct/pattern/alias 기초
- `SiteMesh` defaultdir / excludes 적용
- non-screen `json/file/redirect/action` 분류
- weak inferred card 기본 숨김 정책
- route specificity 기반 정렬

## 현재 약한 영역

### 1. Entry-focused 다양성

- 다중 dispatcher
- 여러 `contextConfigLocation`
- servlet/filter/listener 혼합
- module 경계가 여러 개인 프로젝트

### 2. Persistence 변형

- namespace mismatch
- DAO 이름과 mapper 이름이 어긋나는 구조
- 다중 mapper 후보 충돌
- SQL call 없이 edge만 있는 경우의 오탐 관리

### 3. View / Layout 변형

- nested layout
- 여러 view resolver
- prefix/suffix가 복수인 경우
- JSP alias가 실제 파일 경로와 더 어긋나는 구조

### 4. Real-project Gate

- 익명화된 실제 프로젝트 기준 스냅샷 비교
- 공개 샘플 기반 실패 모드 기록

## 현재 미검증 / 부족한 영역

- 공개 샘플 4차 검증
- representative real-project gate 반복

### Action-family 현재 강한 영역

- Struts2 `struts.xml` action / method / jsp result 복원
- Stripes `@UrlBinding`, `@DefaultHandler`, `@HandlesEvent` 복원
- Stripes JSP `beanclass` / `submit name` 기반 역방향 action 링크 복원
- `ForwardResolution(CONSTANT)`와 `StreamingResolution` 최소 해석
- Struts `redirectAction`과 Stripes self redirect 최소 해석
- Stripes `session.getAttribute("/actions/*.action")` alias를 request path supporting evidence로 보존
- 공개 Stripes 샘플에서 대표 screen flow 복원

### Action-family 현재 약한 영역

- `accountBean` style typed session attribute alias inference의 real-project 검증
- redirect-action / chain 계열 wildcard 세분화
- Struts 1 `struts-config.xml` / form bean / forward chain
- `DynamicMappingFilter` 계열 boundary
