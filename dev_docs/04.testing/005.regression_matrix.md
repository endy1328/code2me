- 현재 `Legacy Java EE` 분석이 어떤 축을 검증 중인지, 어떤 축이 비어 있는지 한 장에서 확인한다.

## 현재 강한 영역

- annotation 기반 request mapping
- 다중 dispatcher별 entry context 선택
- `SimpleUrlHandlerMapping` + method resolver 기반 XML multi-action
- `BeanNameUrlHandlerMapping` 기반 XML-heavy controller
- direct SQL evidence 우선 / fallback 억제
- `SiteMesh` direct/pattern/alias 기초
- `SiteMesh` defaultdir / excludes 적용
- non-screen `json/file/redirect/action` 분류
- weak inferred card 기본 숨김 정책
- route specificity 기반 정렬

## 현재 약한 영역

### 1. Entry-focused 다양성

- 다중 dispatcher
- 여러 `contextConfigLocation`
- servlet/filter/listener 혼합
- module 경계가 여러 개인 프로젝트

### 2. Persistence 변형

- namespace mismatch
- DAO 이름과 mapper 이름이 어긋나는 구조
- 다중 mapper 후보 충돌
- SQL call 없이 edge만 있는 경우의 오탐 관리

### 3. View / Layout 변형

- nested layout
- 여러 view resolver
- prefix/suffix가 복수인 경우
- JSP alias가 실제 파일 경로와 더 어긋나는 구조

### 4. Real-project Gate

- 익명화된 실제 프로젝트 기준 스냅샷 비교
- 공개 샘플 기반 실패 모드 기록

## 현재 미검증 / 부족한 영역

- environment / locale 분기 fixture
- 공개 샘플 2차 검증
- 실제 프로젝트 read-only gate

## 우선순위 확정

### P0

- `legacy-java-ee-mixed-web-api`
  - screen / json / file 혼합 non-screen 분류
  - concrete route ordering
  - visible confirmed data path와 UI action 연결

- representative public sample gate
  - `jpetstore-6` 기준 soft fail 기록 유지
  - framework entry는 잡히지만 controller/action 복원 한계 명시

### P1

- `legacy-java-ee-view-resolver-variants`
  - 여러 resolver
  - logical view -> JSP 후보 확장

- `legacy-java-ee-env-branch`
  - locale / env 분기
  - properties-config 도입 이후 회귀 준비

### P2

- 공개 샘플 2차 검증
  - Struts / Stripes / Java config 기반 실패 모드 축적
  - profile boundary 확정

## 운영 원칙

- 일상 회귀의 1순위는 `합성 fixture`다.
- 공개 샘플은 구조 다양성 확인용으로 사용한다.
- 익명화 실제 프로젝트는 release gate 또는 수동 품질 점검용으로 사용한다.
