# Persona Scenario Regression Cases

## 목적

- 현재 리포트가 어떤 사용자 시나리오를 우선 만족해야 하는지 테스트 관점에서 고정한다.

## Scenario 1. 온보딩 담당자

- 질문:
  - 이 앱은 어떤 URL 패턴으로 요청을 받는가
  - 대표 화면 하나를 따라가면 어떤 controller/service/dao/view로 이어지는가
- 필요한 출력:
  - `Start Here`에 representative framework flow와 screen flow가 보인다
  - `Flow Details`에서 section 순서가 유지된다
  - 화면 JSP의 다음 링크/API action이 `UI Actions`로 이어진다
- 현재 회귀 입력:
  - `legacy-java-ee-minimal`
  - `legacy-java-ee-mixed-web-api`

## Scenario 2. 아키텍처 리뷰어

- 질문:
  - XML mapping과 annotation mapping이 어떤 우선순위로 합쳐지는가
  - direct SQL evidence가 fallback mapper보다 앞서는가
- 필요한 출력:
  - mapping detail에 handler mapping / method resolver / bean 근거가 보인다
  - data access에서 `confirmed` path가 visible이고 fallback은 main flow를 대체하지 않는다
- 현재 회귀 입력:
  - `legacy-java-ee-action-mapping`
  - `legacy-java-ee-bean-name-mapping`
  - `legacy-java-ee-persistence-priority`

## Scenario 3. 운영 이관 담당자

- 질문:
  - 다운로드/API 요청이 screen 흐름과 구분되는가
  - dispatcher와 관련 config를 끝까지 따라갈 수 있는가
- 필요한 출력:
  - non-screen flow에서 `response kind`, `content types`, `file response hints`가 보인다
  - multi-dispatcher일 때 entry pattern과 dispatcher config가 흐름에 붙는다
- 현재 회귀 입력:
  - `legacy-java-ee-entry-multi-dispatcher`
  - `legacy-java-ee-mixed-web-api`

## 회귀 적용 원칙

- 새로운 fixture는 최소 한 persona scenario에 직접 연결돼야 한다.
- scenario에 연결되지 않는 출력은 보조 정보로만 취급하고 primary regression 기준으로 올리지 않는다.
