# 실제 프로젝트 검증 게이트

## 목적

- 합성 fixture 중심 회귀만으로는 놓칠 수 있는 실제 레거시 구조를 검증한다.
- 하지만 일상 개발 속도를 해치지 않도록, 실제 프로젝트 검증은 `release gate` 또는 `정기 수동 점검`으로 분리한다.

## 기본 원칙

- 일상 회귀의 1순위는 `합성 fixture`다.
- 실제 프로젝트 검증은 `정확도 확인용 품질 게이트`다.
- 실제 프로젝트 입력은 가능하면 `read-only snapshot` 또는 `익명화된 구조`로 받는다.
- 민감 정보 제거 과정에서 `class / bean / mapper / route` 연결 신호가 사라지면 검증 가치가 떨어지므로, 익명화 정책은 보수적으로 설계한다.

## 권장 입력 형태

### 1. 1순위

- 익명화된 실제 프로젝트의 `snapshot.json`
- 익명화된 입력 프로젝트 트리
- 최소 실행 로그

### 2. 2순위

- 공개 가능한 구조 유사 샘플
- 문제가 재현되는 축만 축소한 fixture

### 3. 지양

- 권한 제약 때문에 반복 검증이 어려운 원본 사내 저장소를 일상 회귀에 직접 연결하는 방식

## 검증 항목

- profile detection이 기대 범위 안에 들어오는가
- 주요 `web.xml` entry와 dispatcher가 보존되는가
- 대표 screen flow가 최소 1개 이상 설명 가능한가
- 대표 non-screen flow가 최소 1개 이상 분류 가능한가
- controller/service/dao 연결이 과도한 false positive 없이 유지되는가
- mapper/sql 또는 integration path가 과장 없이 유지되는가
- `heuristic` 카드가 main flow를 대체하지 않는가
- `report.html`과 내부 미러 산출물이 정상 생성되는가

## 판정 방식

### Pass

- 핵심 진입 흐름과 대표 business path가 사람 눈으로 추적 가능하다.
- 강한 근거가 있는 flow가 유지된다.
- 약한 추정은 보조 카드로 남는다.

### Soft Fail

- 주요 flow는 보이지만 route 선택, layout 연결, SQL fallback 등에서 일부 오탐/누락이 있다.
- 합성 fixture로 재현 가능한 축으로 환원해서 후속 작업으로 보낸다.

### Hard Fail

- 대표 entry가 누락된다.
- screen/api가 뒤섞인다.
- direct evidence보다 fallback이 앞선다.
- 내부 미러나 report generation이 깨진다.

## 운영 절차

1. 실제 프로젝트 또는 익명화 snapshot을 확보한다.
2. `npm run analyze -- <project-root>` 또는 snapshot 비교 절차를 수행한다.
3. 핵심 flow 3개를 수동 확인한다.
4. 실패 패턴을 `합성 fixture`로 환원한다.
5. 회귀 테스트를 추가한 뒤 실제 프로젝트 게이트를 다시 확인한다.

## 최근 공개 샘플 게이트 기록

### 2026-04-13: `mybatis/jpetstore-6`

- 입력 형태:
  - 공개 GitHub 샘플 프로젝트 clone
  - `web.xml`, `applicationContext.xml`, JSP 포함
- 실행:
  - `npm run analyze -- /tmp/jpetstore-6-code2me`
- 결과:
  - `profileDetection.score = 9`
  - `entryPointCount = 1`
  - `warningCount = 0`
  - `StripesDispatcher (*.action)` entry는 보존
  - controller/action 복원은 실패하고 view-only flow가 다수 생성
- 판정:
  - `Soft Fail`
- 실패 분류:
  - `mapping`
  - `profile boundary`
- 해석:
  - 현재 `legacy-java-ee` 프로파일은 `web.xml + Spring XML + JSP` 신호만으로 매치되지만, Stripes action 계열은 handler 복원 규칙이 없다.
  - profile detection은 맞았지만 request mapping 해석은 현재 범위 밖이다.
- 후속 조치:
  - 다음 profile 후보 문서에 `Struts/Stripes action-family`를 별도 후보로 유지
  - `legacy-java-ee` 문서에는 `Spring MVC XML/annotation 우선` 범위를 더 명확히 적는다
