# Legacy Java EE 최소 어댑터 세트

## 문서 목적

이 문서는 `Legacy Java EE 프로파일`을 실제로 구현하기 위해 필요한 최소 어댑터 세트를 정의한 문서다.

목표는 아래를 명확히 하는 것이다.

- 첫 구현에서 어떤 어댑터를 반드시 만들 것인가
- 각 어댑터의 책임이 어디까지인가
- 어댑터 간 연결 순서는 어떻게 되는가
- 어떤 결과를 만들면 최소 가치가 성립하는가

## 최소 세트 선정 원칙

- 온보딩에 직접 기여하는 정보부터 추출한다.
- 구조와 진입점 파악에 필요한 어댑터를 먼저 둔다.
- 범용 코어에 재사용 가능한 단위로 쪼갠다.
- 첫 구현에서 과도한 정적 분석은 피한다.

## 최소 필수 어댑터 목록

첫 구현에서 필수로 보는 어댑터는 아래 네 개다.

1. `ant-build-xml`
2. `web-xml`
3. `spring-xml`
4. `java-source-basic`

이 네 개가 있으면 최소한 아래를 만들 수 있다.

- 프로젝트/모듈 구조
- 배포 단위 기본 정보
- 웹 진입점 후보
- Controller/Service/DAO 구조 후보

## 1. ant-build-xml

### 목적

Ant 기반 빌드 구조를 읽고 모듈 경계와 배포 단위를 파악한다.

### 주요 입력

- `**/build.xml`
- import 된 build 관련 xml
- 참조 property 파일

### 주요 출력 엔터티

- `project`
- `module`
- `build_target`
- `deployment_unit`

### 주요 출력 관계

- `contains`
- `depends_on`
- `deploys`

### 핵심 가치

- 어떤 모듈이 존재하는지
- 어떤 타깃이 EAR/WAR/JAR 생성과 연결되는지
- 공통 라이브러리와 웹 모듈 경계를 어디서 볼지

## 2. web-xml

### 목적

전통 웹 진입 구조의 시작점인 `web.xml` 을 읽어 servlet/filter/listener 와 초기 설정 흐름을 추출한다.

### 주요 입력

- `**/web.xml`

### 주요 출력 엔터티

- `config`
- `deployment_unit`
- `route`

### 주요 출력 관계

- `configures`
- `routes_to`
- `references`

### 핵심 가치

- 어떤 모듈이 웹 진입을 가지는지
- 어떤 dispatcher servlet 또는 context 설정이 연결되는지
- 웹 진입점의 1차 후보가 무엇인지

## 3. spring-xml

### 목적

Spring MVC XML 설정을 읽어 controller, bean, view resolver, context 연결 구조를 추출한다.

### 주요 입력

- `**/*dispatcher*.xml`
- `**/applicationContext*.xml`

### 주요 출력 엔터티

- `config`
- `controller`
- `service`
- `view`

### 주요 출력 관계

- `configures`
- `declares`
- `routes_to`
- `references`

### 핵심 가치

- controller bean 후보 식별
- view resolver 및 context 구조 파악
- `web.xml` 에서 시작된 설정 흐름을 실제 스프링 구성으로 연결

## 4. java-source-basic

### 목적

Java 소스 파일을 스캔해 class 단위 구조와 기본 역할 후보를 추출한다.

### 주요 입력

- `**/*.java`

### 주요 출력 엔터티

- `file`
- `class`
- `controller`
- `service`
- `dao`

### 주요 출력 관계

- `contains`
- `implements`
- `extends`
- `depends_on`
- `calls`

### 핵심 가치

- Controller/Service/DAO 성격 클래스 파악
- 공통 라이브러리 중심 구조 파악
- 이후 mapper 연결을 위한 Java 측 기준점 생성

## 1차 확장 어댑터 목록

최소 세트 이후 바로 붙이기 좋은 어댑터는 아래와 같다.

- `ibatis-sql-map`
- `mybatis-mapper`
- `jsp-view`
- `sitemesh-config`

이 어댑터들이 붙으면 온보딩 가치는 크게 올라간다.

- DAO 와 SQL 연결
- controller 와 JSP 연결
- 공통 레이아웃 구조 설명

## 5. ibatis-sql-map

### 목적

iBATIS 2.x 계열 SQL map XML 을 읽어 SQL statement 와 namespace 정보를 추출한다.

### 주요 출력 엔터티

- `mapper`
- `sql_statement`

### 주요 출력 관계

- `contains`
- `queries`
- `maps_to`

## 6. mybatis-mapper

### 목적

MyBatis 3.x mapper XML 을 읽어 mapper namespace 와 statement 를 추출한다.

### 주요 출력 엔터티

- `mapper`
- `sql_statement`

### 주요 출력 관계

- `contains`
- `queries`
- `maps_to`

## 7. jsp-view

### 목적

JSP 파일과 view name 추론을 통해 controller 결과와 화면 파일을 연결한다.

### 주요 출력 엔터티

- `view`
- `file`

### 주요 출력 관계

- `renders`
- `references`

## 8. sitemesh-config

### 목적

SiteMesh 설정을 읽어 공통 레이아웃과 decorator 구조를 파악한다.

### 주요 출력 엔터티

- `config`
- `view`

### 주요 출력 관계

- `configures`
- `renders`

## 어댑터 간 의존성

권장 의존성은 아래와 같다.

- `web-xml` 는 `ant-build-xml` 이후 실행하는 편이 좋다.
- `spring-xml` 은 `web-xml` 결과를 참조할 수 있다.
- `java-source-basic` 은 독립 실행 가능하지만 `spring-xml` 과 병합 가치가 높다.
- `ibatis-sql-map`, `mybatis-mapper` 는 `java-source-basic` 결과와 연결될 때 가치가 높다.
- `jsp-view` 는 `spring-xml` 또는 Java controller 결과와 함께 쓰일 때 가치가 높다.

## 권장 실행 순서

1. `ant-build-xml`
2. `web-xml`
3. `spring-xml`
4. `java-source-basic`
5. `ibatis-sql-map`, `mybatis-mapper`
6. `jsp-view`, `sitemesh-config`

## 최소 가치 기준

Legacy Java EE 프로파일의 첫 구현은 아래가 되면 최소 가치를 달성한 것으로 볼 수 있다.

- 웹 모듈과 공통 라이브러리 모듈을 구분할 수 있다.
- `web.xml` 에서 시작되는 대표 웹 진입 구조를 보여줄 수 있다.
- 대표 controller 와 service/dao 흐름을 보여줄 수 있다.
- 공통 라이브러리 의존을 드러낼 수 있다.

## 일부러 제외하는 것

첫 구현에서 아래는 최소 세트에 넣지 않는 편이 좋다.

- 메서드 수준 정밀 call graph
- 완전한 Spring bean wiring 재현
- 완전한 ORM/SQL 실행 경로 복원
- 모든 운영환경 설정 정밀 추적

## 구현 우선순위
