# 프로파일 중심 제품 정의

## 문서 목적

이 문서는 code2me의 제품 정의를 `언어 중심 지원`에서 `분석 프로파일 중심 지원`으로 재정리한 기준 문서다.

이 문서의 목표는 아래를 명확히 하는 것이다.

- 제품을 어떤 방식으로 정의할 것인가
- 사용자가 무엇을 입력하고 무엇을 결과로 받는가
- 분석 프로파일이 제품 안에서 어떤 역할을 하는가
- 향후 여러 언어와 프로젝트 타입을 어떤 구조로 확장할 것인가

## 제품 정의

code2me는 `프로젝트 타입별 분석 프로파일`을 기반으로 소스 구조를 해석하고, 신규 입사 개발자의 온보딩에 필요한 구조, 진입점, 관계, 설명 정보를 제공하는 코드 지식 도구다.

## 한 줄 구조

제품의 기본 구조는 아래와 같다.

- 입력: 분석 대상 소스
- 해석 기준: 분석 프로파일
- 출력: 프로파일에 맞춘 분석 결과

즉, 제품은 단순히 특정 언어를 지원하는 도구가 아니라, 소스의 성격에 맞는 해석 규칙 세트를 적용하는 도구다.

## 왜 프로파일 중심이 맞는가

같은 언어라도 프로젝트 타입이 다르면 구조를 읽는 방식이 크게 달라진다.

예를 들어 Java만 보더라도 아래는 서로 다른 분석 규칙이 필요하다.

- Spring Boot 단일 서비스
- Legacy Java EE 시스템
- 배치 중심 Java 애플리케이션

TypeScript/JavaScript도 마찬가지다.

- Next.js 애플리케이션
- React SPA
- Node.js 라이브러리 모노레포

따라서 제품의 1차 개념을 `언어`로 두기보다 `분석 프로파일`로 두는 편이 정확하다.

## 분석 프로파일의 정의

분석 프로파일은 특정 프로젝트 타입에 대해 아래를 한 묶음으로 정의한 규칙 세트다.

- 우선 읽을 입력 소스
- 진입점 탐색 규칙
- 관계 추출 규칙
- 중요하게 볼 계층
- 결과 화면에서 강조할 정보
- 예외 처리 또는 오탐 방지 규칙

즉, 프로파일은 단순 파서 선택이 아니라 `이 프로젝트를 어떻게 이해할 것인가`에 대한 제품 규칙이다.

## 제품 요구사항 문장

### 핵심 요구사항

- 사용자는 분석 대상 소스를 등록할 수 있어야 한다.
- 사용자는 분석 프로파일을 직접 선택하거나 추천받을 수 있어야 한다.
- 시스템은 선택된 프로파일에 맞는 입력 소스를 우선 수집해야 한다.
- 시스템은 프로파일 규칙에 따라 구조, 진입점, 파일/모듈 관계를 생성해야 한다.
- 시스템은 자동 분석 결과 위에 사람이 설명을 보강할 수 있어야 한다.
- 시스템은 원본 소스를 수정하지 않아야 한다.
- 시스템은 협업 입력을 허용하되 수정 이력을 남겨야 한다.

### 출력 요구사항

분석 결과는 최소한 아래를 제공해야 한다.

- 프로젝트 구조
- 패키지/폴더/파일 관계
- 온보딩 시작점
- 중요 설정 및 진입점
- 사람이 보강한 설명

## 제품 내부 구조 제안

프로파일 중심 구조는 아래와 같이 나눌 수 있다.

### 1. Source Intake

- 저장소 또는 소스 입력
- 기본 파일 인덱싱
- 파일 타입 분류

### 2. Profile Detector

- 소스 구조를 보고 프로파일 후보 추천
- 사용자가 직접 프로파일 선택 가능
- 오탐 가능성이 높은 경우 후보 여러 개 제시

### 3. Profile Definition

프로파일별로 아래 정보를 가진다.

- 입력 소스 패턴
- 진입점 규칙
- 관계 규칙
- 표시 우선순위
- 예외 규칙

### 4. Analysis Engine

- 선택된 프로파일에 맞춰 분석 실행
- 구조 그래프와 메타데이터 생성
- 프로파일별 특화 관계 생성

### 5. Result Presenter

- 프로파일에 맞는 설명 방식으로 결과 제공
- 시작점, 흐름, 중요 모듈, 설정 파일 강조

## 현재 합의와의 연결

지금까지 정리된 내용은 이 구조 안에서 자연스럽게 연결된다.

- 제품 목적: 신규 입사 개발자 온보딩
- 분석 방식: 자동 분석 + 수동 보강
- 메타데이터 저장: 프로그램 내부 저장
- 원본 소스: 수정하지 않음
- 협업 방식: 모든 개발자 입력 가능
- 변경 관리: 수정 이력만 관리

즉, 프로파일 중심 구조는 기존 합의를 대체하는 것이 아니라 더 정확한 상위 구조로 묶는 역할을 한다.

## 현재 우선 프로파일 예시

현재 논의 기준으로 아래와 같은 프로파일이 존재할 수 있다.

- `Legacy Java EE 프로파일`
- `TS/JS Monorepo 프로파일`
- `Spring Boot 프로파일`
- `Python 애플리케이션 프로파일`

이 중 현재 실제 우선 적용 대상은 `Legacy Java EE 프로파일`이다.

## 현재 실제 프로젝트에 대한 적용 방식

현재 먼저 처리해야 하는 Java 시스템은 아래처럼 이해하면 된다.

- 입력: Java 8 + Ant + EAR/WAR + Spring MVC XML + JSP + iBATIS/MyBatis 혼합 소스
- 프로파일: `Legacy Java EE`
- 출력:
  - `web.xml`, Spring XML, Controller, Service, DAO, Mapper 연결 구조
  - shared-lib 중심 비즈니스 흐름
  - JSP/SiteMesh 기반 화면 진입 구조
  - 운영환경 및 배포 구조 설명

## 향후 확장 방식

새 언어나 프레임워크를 추가할 때는 제품 구조를 바꾸기보다 프로파일을 추가하는 방식으로 확장한다.

즉, 확장 단위는 아래와 같다.

- 새 언어 추가
- 새 프레임워크 추가
- 새 프로젝트 타입 추가

하지만 제품 내부에서는 모두 `새 분석 프로파일 추가`로 취급한다.

## 범위 제어 원칙

프로파일 중심 설계를 택하더라도 첫 버전에서 모든 프로파일을 동시에 고도화하면 범위가 무너진다.

따라서 아래 원칙을 유지한다.

- 한 번에 하나의 우선 프로파일을 정한다.
- 프로파일별 MVP를 먼저 만든다.
- 프로파일 간 공통 메타모델은 최대한 재사용한다.
- 프로파일별 예외 규칙은 별도로 유지한다.

## 권장 제품 문장

아래 문장을 현재 기준 제품 설명으로 사용할 수 있다.

`code2me는 프로젝트 타입별 분석 프로파일을 기반으로 소스를 해석하고, 신규 입사 개발자가 구조와 진입점을 빠르게 이해할 수 있도록 돕는 온보딩용 코드 지식 도구다.`
