# 결과 병합 규칙 초안

## 문서 목적

이 문서는 여러 어댑터가 만든 결과를 공통 메타모델 위에서 어떻게 병합할지에 대한 초안을 정리한 문서다.

목표는 아래를 명확히 하는 것이다.

- 서로 다른 어댑터가 같은 대상을 가리킬 때 어떻게 합칠 것인가
- 어떤 경우 새 엔터티를 만들고 어떤 경우 기존 엔터티에 붙일 것인가
- 신뢰도와 출처를 어떻게 유지할 것인가
- Legacy Java EE 같은 혼합 구조에서 어떤 병합 규칙이 핵심인가

## 왜 병합 규칙이 필요한가

범용 분석기에서는 하나의 대상이 여러 어댑터 결과에 동시에 등장한다.

예:

- `spring-xml` 어댑터가 controller bean 을 찾는다.
- `java-source-basic` 어댑터가 controller 클래스를 찾는다.
- `web-xml` 어댑터가 해당 controller 로 이어지는 설정 흐름을 찾는다.

이 세 결과를 따로 저장하면 사용자는 중복된 객체를 보게 되고, 흐름도 분리된다.

따라서 병합 규칙은 코어 품질의 핵심이다.

## 병합 설계 원칙

- 어댑터 결과는 사실 기반 조각으로 본다.
- 병합은 코어에서 일관되게 처리한다.
- 무리한 병합보다 보수적 병합이 낫다.
- 병합 후에도 출처와 근거는 잃지 않아야 한다.
- 불확실한 연결은 낮은 신뢰도로 유지한다.

## 병합 대상 유형

병합은 크게 아래 네 종류에서 일어난다.

- 엔터티 병합
- 관계 병합
- 진입점 병합
- 설명/수동 보강 병합

## 1. 엔터티 병합 규칙

엔터티 병합은 서로 다른 어댑터가 같은 실체를 가리킬 때 수행한다.

주요 병합 키 후보는 아래와 같다.

- 절대 경로 또는 상대 경로
- 정규화된 논리 이름
- 클래스 FQN
- mapper namespace
- sql statement id
- 설정 파일 경로 + bean id 조합

## 엔터티 병합 우선순위

병합 판단은 아래 순서를 권장한다.

1. 강한 식별자 일치
2. 경로 + 유형 일치
3. 이름 + 컨텍스트 일치
4. 추론 기반 후보 일치

### 1. 강한 식별자 일치

아래는 거의 확실한 병합 기준이다.

- 동일 파일 경로
- 동일 Java FQN
- 동일 mapper namespace
- 동일 statement id 와 동일 mapper 파일

### 2. 경로 + 유형 일치

예:

- 같은 `applicationContext.xml` 을 두 어댑터가 각각 config 엔터티로 만들었을 때

### 3. 이름 + 컨텍스트 일치

예:

- 같은 모듈 안의 `OrderService`
- 같은 package 안의 `UserDao`

### 4. 추론 기반 후보 일치

예:

- DAO 클래스 이름과 mapper namespace 이름이 유사할 때
- JSP view name 과 실제 파일 경로가 규칙적으로 대응할 때

이 단계는 낮은 신뢰도로 처리하는 편이 좋다.

## 엔터티 병합 결과 규칙

병합이 일어나면 아래 정보를 유지해야 한다.

- 대표 엔터티 ID
- 병합된 원본 후보 ID 목록
- 기여 어댑터 목록
- 증거 목록
- 최종 신뢰도

즉, 병합은 정보를 버리는 것이 아니라 통합하는 것이다.

## 2. 관계 병합 규칙

서로 다른 어댑터가 사실상 같은 관계를 발견할 수 있다.

예:

- `spring-xml` 이 controller 와 view resolver 연결을 잡는다.
- `jsp-view` 가 controller 와 JSP 후보 연결을 잡는다.

관계 병합 기본 키는 아래 조합이 적절하다.

- `type`
- `from`
- `to`
- 관계 메타데이터의 핵심 키

동일 관계로 판단되면:

- `sourceAdapterIds` 를 합친다.
- `confidence` 는 더 높은 쪽을 우선하되 근거를 누적한다.
- 상충 메타데이터는 evidence 로 남긴다.

## 3. 진입점 병합 규칙

진입점은 여러 어댑터가 같은 시작점을 다르게 발견할 수 있다.

예:

- `web-xml` 이 servlet 진입을 찾음
- `spring-xml` 이 handler mapping 기반 진입을 찾음
- `java-source-basic` 이 controller 클래스를 찾음

병합 기준:

- 같은 target entity
- 같은 URL 또는 같은 servlet context
- 같은 모듈

병합 후에는 아래를 남기는 편이 좋다.

- 진입점 제목
- 진입 근거 목록
- 우선순위
- 신뢰도

## 4. 설명과 수동 보강 병합 규칙

자동 분석 결과와 수동 입력은 덮어쓰기보다 공존이 원칙이다.

권장 원칙:

- 자동 설명과 수동 설명을 별도 보관
- 기본 표시 우선순위는 수동 설명 우선
- 자동 설명은 근거/초안 역할 유지
- 변경 이력은 항상 남김

즉, 병합은 `manual overwrites auto` 라기보다 `manual preferred, auto preserved` 가 낫다.

## 병합 신뢰도 규칙

병합 후 신뢰도는 아래 기준으로 산정하는 편이 좋다.

- 강한 식별자 일치: `high`
- 경로/유형 일치: `high`
- 이름/컨텍스트 일치: `medium`
- 추론 기반 매칭: `low`

여러 어댑터가 같은 결론을 지지하면 신뢰도를 올릴 수 있다.

예:

- `spring-xml` + `java-source-basic` 가 같은 controller 를 지지하면 `high`
- 이름 유사성만으로 연결되면 `low`

## 출처 유지 규칙

병합 후에도 아래 정보는 반드시 남기는 편이 좋다.

- 어떤 어댑터가 기여했는가
- 어떤 파일에서 근거가 나왔는가
- 어떤 규칙으로 병합되었는가
- 자동 병합인지 수동 보강인지

즉, 병합은 결과를 단순하게 만들지만 추적성은 잃지 않아야 한다.

## Legacy Java EE에서 중요한 병합 사례

### 사례 1. web.xml + spring-xml

- `web.xml` 이 dispatcher servlet 을 설정
- `spring-xml` 이 해당 servlet context 안의 controller bean 을 발견
- 병합 결과:
  - 웹 진입 흐름 생성
  - `configures`, `routes_to` 관계 강화

### 사례 2. spring-xml + java-source-basic

- `spring-xml` 이 bean class 이름을 발견
- `java-source-basic` 이 실제 class 엔터티를 발견
- 병합 결과:
  - controller/service/dao 역할 확정
  - config 와 class 연결 강화

### 사례 3. java-source-basic + mybatis/ibatis

- Java DAO 클래스 발견
- mapper namespace / statement 발견
- 병합 결과:
  - `dao -> mapper -> sql_statement` 흐름 형성

### 사례 4. spring-xml + jsp-view + sitemesh

- controller 가 view name 을 반환
- JSP 어댑터가 실제 파일 후보를 찾음
- SiteMesh 어댑터가 decorator 구조를 찾음
- 병합 결과:
  - `controller -> view -> layout` 흐름 형성

## 병합 충돌 규칙

서로 충돌하는 결과가 나올 수 있다.

예:

- 같은 bean 이름이 다른 클래스를 가리킴
- DAO 와 mapper 연결 후보가 둘 이상
- view name 이 여러 JSP 후보와 매칭

권장 처리:

- 강한 식별자 없는 경우 자동 확정하지 않음
- 다중 후보를 유지하고 신뢰도 낮게 표시
- 필요하면 수동 보강 대상 큐에 올림

## 병합 단계 구조

분석 엔진은 병합을 아래 단계로 나누는 편이 좋다.

1. 엔터티 정규화
2. 엔터티 병합
3. 관계 재매핑
4. 진입점 병합
5. 신뢰도 재계산
6. 설명/수동 보강 결합

## 범용 분석기 관점의 결론

병합 규칙은 특정 프로파일 구현 디테일이 아니라 범용 분석기 코어의 핵심 기능이다.

어댑터는 조각을 만들고, 코어는 병합 규칙으로 그 조각을 하나의 탐색 가능한 지식 그래프로 만든다.
