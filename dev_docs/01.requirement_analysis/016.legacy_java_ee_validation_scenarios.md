# Legacy Java EE 검증 시나리오 초안

## 문서 목적

이 문서는 `Legacy Java EE 프로파일`의 첫 구현 결과를 어떤 기준으로 검증할지 정리한 문서다.

## 검증 목표

아래 항목이 실제로 동작하는지 확인한다.

- 구조 파악
- 진입점 탐색
- `Controller -> Service -> DAO` 연결
- `DAO -> Mapper -> SQL` 연결
- `Controller -> View -> Layout` 연결
- 결과 병합 규칙 반영
- 공통 라이브러리 중심축 표시

## 검증 방식

검증은 아래 순서로 수행하는 편이 좋다.

1. 어댑터 단위 검증
2. 병합 검증
3. 통합 검증

## 대표 통과 흐름

### 구조 흐름

- `build.xml`을 읽었을 때 모듈과 배포 단위를 식별할 수 있어야 한다.

### 웹 진입 흐름

- `web.xml`을 읽었을 때 servlet, filter, listener와 진입점 후보를 식별할 수 있어야 한다.
- `dispatcherServlet.xml` 또는 `applicationContext*.xml`을 읽었을 때 controller bean, view resolver, context 연결을 식별할 수 있어야 한다.

### Java 흐름

- Java 소스를 읽었을 때 controller, service, dao 후보를 식별할 수 있어야 한다.

### persistence 흐름

- iBATIS/MyBatis XML을 읽었을 때 dao와 SQL statement가 연결되어야 한다.

### view 흐름

- JSP/SiteMesh를 읽었을 때 `controller -> view -> layout` 흐름이 보여야 한다.

### 병합 흐름

- 병합 후에는 같은 대상을 가리키는 결과가 중복 노드가 아니라 하나의 흐름으로 보정되어야 한다.

### 공통 라이브러리 흐름

- `shared-lib`가 공통 비즈니스 중심축으로 드러나야 한다.

## 샘플 프로젝트 체크리스트

- `build.xml`이 존재하는가
- `web.xml`이 존재하는가
- `dispatcherServlet.xml` 또는 유사 Spring XML이 존재하는가
- `applicationContext*.xml`이 존재하는가
- `*.java`에서 controller/service/dao 역할이 확인되는가
- `sql-map*.xml` 또는 `*mapper.xml`이 존재하는가
- `*.jsp`가 존재하는가
- `sitemesh` 관련 설정이 존재하는가
- `shared-lib` 같은 공통 라이브러리 모듈이 식별되는가
- 지역/국가별 설정 파일이 존재하는가
- 운영환경별 설정 파일이 존재하는가
- 배포 단위가 EAR/WAR 관점으로 설명되는가

## 작업 단위별 검증 포인트

### 최소 어댑터 4종

- `ant-build-xml`: 모듈과 배포 단위를 식별하는가
- `web-xml`: 진입 servlet/filter/listener를 식별하는가
- `spring-xml`: controller bean, view resolver, context를 식별하는가
- `java-source-basic`: controller/service/dao 후보를 식별하는가

### 병합 로직

- 중복 엔터티가 하나로 합쳐지는가
- 관계가 끊기지 않는가
- 낮은 신뢰도 추론을 구분해서 표시하는가

### 확장 어댑터

- mapper/sql 연결이 형성되는가
- jsp/layout 연결이 형성되는가

## 판정 기준

### 통과

- 대표 흐름이 최소 1개 이상 실제로 연결된다.
- 중복과 충돌이 제어된다.
- 설명 가능한 온보딩 구조가 만들어진다.

### 부분 통과

- 구조는 보이지만 mapper/view 병합이나 시작점 후보가 불완전하다.

### 실패

- Spring Boot처럼 오해한다.
- `web.xml -> Spring XML -> Controller` 축이 안 잡힌다.
- DAO/mapper 연결이 끊긴다.

## 검증 에이전트 출력 권장 형식

- 검증 대상
- 확인한 흐름
- 누락된 흐름
- 중복/충돌 여부
- 판정
- 후속 조치

## 검증 완료 기준

- 첫 구현 기준 필수 흐름이 문서화된 방식대로 재현된다.
- 샘플 프로젝트 체크리스트가 통과된다.
- 후속 확장 어댑터가 어디를 메울지 명확해진다.
