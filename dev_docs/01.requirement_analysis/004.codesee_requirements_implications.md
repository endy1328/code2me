# CodeSee 분석을 바탕으로 한 요구사항 시사점

## 문서 목적

이 문서는 CodeSee 조사 결과를 "비슷한 프로그램을 실제로 개발하려면 무엇이 필요하나"라는 요구사항 관점으로 전환한 문서다.

## 먼저 결론

CodeSee와 비슷한 프로그램을 만들려면 단순 코드 시각화 도구로 접근하면 안 된다. 최소한 아래 네 가지를 함께 설계해야 한다.

- 코드 구조를 자동 생성하는 분석 엔진
- 구조를 이해 가능한 형태로 보여주는 상호작용 UI
- 변경/리뷰/온보딩에 연결되는 워크플로우
- 사람이 지식을 덧입히고 공유하는 협업 레이어

## 문제 정의 제안

초기 제품은 아래 문제 중 하나를 명확히 1순위로 잡아야 한다.

1. 대형 PR 리뷰 시간을 줄이고 싶다
2. 레거시 코드베이스 온보딩 시간을 줄이고 싶다
3. 멀티리포/서비스 구조 파악 시간을 줄이고 싶다

이 셋을 동시에 정면으로 풀면 너무 무거워진다. CodeSee는 결과적으로 넓은 범위를 다루지만, 실제 개발은 하나의 진입점부터 잡는 편이 낫다.

## MVP 권장 범위

가장 현실적인 MVP는 아래 조합이다.

### 필수

- 단일 저장소 대상 코드베이스 맵
- 파일/디렉터리 간 의존 관계 표시
- PR 기준 영향 맵 생성
- 노드 선택 시 코드/메타데이터 열람
- 필터링 및 포커스 뷰

### 강력 권장

- 라벨과 노트
- 리뷰 순서나 온보딩 순서를 지정할 수 있는 투어
- 저장소 변경 시 증분 재생성
- 맵 정확도 문제를 보정하는 설정 파일

### 후순위

- 서비스 맵
- APM / OpenTelemetry 연동
- 자동화 트리거 엔진
- AI 요약/투어 생성

## 기능 우선순위 제안

### P0

- 저장소 수집
- 언어 파서 또는 의존성 추출기
- 그래프 모델
- 그래프 레이아웃
- 웹 기반 맵 UI
- PR 부분 그래프 생성

### P1

- 라벨/노트/코멘트
- 저장/공유 가능한 뷰
- 협업 권한
- 증분 업데이트 파이프라인
- 설정 파일 기반 정확도 보정

### P2

- IDE 연동
- AI 요약
- 코드 변경 규칙 자동화
- 서비스/런타임 맵

## 권장 데이터 모델

초기에는 아래 정도의 범용 노드/엣지 모델로 시작하는 편이 안전하다.

- 노드: repository, directory, file, symbol, service, database, api endpoint, note, label
- 엣지: contains, imports, calls, depends_on, changed_with, owns, annotated_by, traces_to

이렇게 해두면 나중에 Review Map, Service Map, AI 요약, 자동화 규칙을 같은 그래프 위에 얹기 쉽다.

## 권장 시스템 구조

### 1. Analysis pipeline

- 저장소 수집
- 언어별 분석
- 그래프 생성
- 변화 감지

### 2. Graph platform

- 그래프 저장소
- 질의 계층
- 부분 그래프 생성기

### 3. Product surfaces

- codebase map
- review map
- annotations
- shared views

### 4. Integrations

- GitHub/GitLab/Bitbucket
- IDE extension
- 추후 APM/OTel

## 가장 어려운 리스크

### 1. 정확도 리스크

맵이 틀리면 제품 전체 신뢰가 무너진다. 따라서 화려한 UI보다 정확도 검증 체계가 먼저다.

### 2. 스케일 리스크

대규모 저장소에서 노드 수와 엣지 수가 급증하면 레이아웃, 렌더링, 탐색 경험이 급격히 나빠질 수 있다.

### 3. 언어/빌드 다양성 리스크

현실의 저장소는 단일 언어/단일 빌드 도구가 아니다. 자동 추론 실패를 보완하는 설정 체계가 꼭 필요하다.

### 4. 가치 전달 리스크

맵이 있어도 사용자가 "그래서 뭘 해야 하지?"를 모르면 재방문하지 않는다. 리뷰 투어, 시작 지점 추천, 포커스 뷰 같은 안내 경험이 필요하다.

## 차별화 전략 제안

CodeSee를 그대로 복제하기보다 아래 중 하나를 먼저 강화하는 편이 낫다.

- 리뷰 특화: 대형 PR 영향 분석, 리뷰 순서 추천, 위험 변경 강조
- 온보딩 특화: 팀 지식 맵, 학습 경로, ownership/도메인 뷰
- 아키텍처 특화: 서비스/모듈/데이터 흐름 일관 시각화
- AI 특화: 맵 기반 질의응답, 설명 생성, 변경 영향 해설

## 구현 전략 제안

### 1단계

단일 저장소, 1~2개 언어, 파일 수준 그래프, PR 영향 맵까지

### 2단계

심볼/함수 수준 분석, 라벨/노트/투어, 증분 업데이트

### 3단계

멀티리포, 권한, 협업, 규칙 자동화

### 4단계

서비스 맵, 관측성 연동, AI

## 설계 원칙

- 자동 생성이 기본이어야 한다
- 수동 보정 경로가 반드시 있어야 한다
- 그래프 하나를 여러 경험에 재사용해야 한다
- 시각화보다 작업 흐름 연결이 더 중요하다
- 정확도와 최신성이 기능 수보다 우선이다

## 즉시 후속 작업 제안

다음 조사 또는 설계 문서에서 바로 이어서 정리할 항목은 아래와 같다.

- 어떤 사용자 시나리오를 1순위로 삼을지 결정
- 지원 언어/프레임워크 후보 정리
- 코드 그래프 데이터 모델 초안 작성
- PR 영향 분석 알고리즘 초안 작성
- UI 정보 구조 와이어프레임 작성
- 정확도 검증용 샘플 저장소 선정
