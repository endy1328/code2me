# Legacy Java EE 프로파일 상세 명세 초안

## 문서 목적

이 문서는 code2me의 현재 우선 적용 대상인 `Legacy Java EE 프로파일`을 실제 구현 가능한 수준으로 상세화한 명세 초안이다.

목표는 아래를 명확히 하는 것이다.

- 어떤 프로젝트를 이 프로파일로 분류할 것인가
- 어떤 입력 소스를 어떤 순서로 해석할 것인가
- 어떤 어댑터를 최소 구성으로 사용할 것인가
- 어떤 엔터티와 관계를 우선 생성할 것인가
- 어떤 결과를 온보딩 관점에서 우선 보여줄 것인가

## 대상 프로젝트 정의

이 프로파일은 아래 성격을 가진 전통적 엔터프라이즈 Java 시스템을 대상으로 한다.

- Java 8 전후
- Ant 기반 빌드
- EAR/WAR 배포 구조
- `web.xml` 기반 웹 애플리케이션
- Spring MVC + XML application context
- JSP 기반 화면
- SiteMesh 기반 레이아웃 가능성
- iBATIS/MyBatis 기반 XML SQL 매핑
- 전통 WAS 배포 방식
- 공통 라이브러리 + 복수 웹 모듈/배치 모듈 구조

## 탐지 신호

### 강한 신호

- `**/build.xml`
- `**/web.xml`
- `**/applicationContext*.xml`
- `**/*.jsp`

### 보조 신호

- `**/*dispatcher*.xml`
- `**/*sql-map*.xml`
- `**/*mapper.xml`
- `**/*sitemesh*.xml`
- JEUS/WebLogic 관련 설정 흔적

### 제외 또는 주의 신호

- `@SpringBootApplication` 만으로 Spring Boot 로 단정하지 말 것
- `application.yml` 존재만으로 현대 Spring 구조로 해석하지 말 것

## 프로파일 가드레일

- Spring Boot 프로젝트로 가정하지 않는다.
- annotation 만으로 진입점을 찾지 않는다.
- XML 설정과 배포 서술자를 우선 해석한다.
- 웹 모듈만 보고 시스템 구조를 단정하지 않는다.
- DAO 와 XML mapper 를 분리된 세계로 보지 않는다.

## 최소 필수 어댑터

첫 구현에서 반드시 필요한 어댑터는 아래와 같다.

- `ant-build-xml`
- `web-xml`
- `spring-xml`
- `java-source-basic`

이 네 개가 있어야 최소한 아래가 가능하다.

- 모듈 구조 파악
- 웹 진입점 후보 파악
- 기본 bean/config 구조 파악
- Controller/Service/DAO 후보 파악

## 1차 선택 어댑터

초기 가치가 큰 선택 어댑터는 아래와 같다.

- `ibatis-sql-map`
- `mybatis-mapper`
- `jsp-view`
- `sitemesh-config`

이 어댑터들이 붙으면 아래 가치가 생긴다.

- DAO 와 SQL 연결
- 화면 렌더링 흐름 설명
- 레이아웃 구조 표시

- `properties-config`

- `batch-job-detect`

## 실행 순서 규칙

권장 실행 순서는 아래와 같다.

1. `ant-build-xml`
2. `web-xml`
3. `spring-xml`
4. `java-source-basic`

## 진입점 탐색 규칙

Legacy Java EE 프로파일의 진입점 탐색은 아래 순서를 따른다.

1. `web.xml` 에서 servlet, filter, listener 확인
2. servlet init-param 또는 context 설정 참조 확인
3. `dispatcherServlet.xml` 또는 관련 Spring XML 로 이동
4. handler mapping, controller bean, view resolver 파악
5. 이후 Java 소스에서 controller / service / dao 흐름 연결
6. JSP 와 SiteMesh 구조 연결

## 핵심 관계 해석 규칙

### 웹 진입 흐름

- `web.xml` 이 servlet context 를 `configures`
- servlet config 가 controller 계층으로 `routes_to`

### 비즈니스 흐름

- controller 는 service 를 `calls`
- service 는 dao 를 `calls`

### persistence 흐름

- dao 는 mapper 또는 sql_statement 를 `queries`
- mapper 는 sql_statement 를 `contains` 또는 `declares`

### 화면 흐름

- controller 는 view 를 `renders`
- SiteMesh 설정은 view 레이아웃을 `configures`

## 공통 라이브러리 우선 해석 규칙

공통 라이브러리 예: `shared-lib`

이 모듈은 단순 보조 모듈이 아니라 비즈니스 중심축으로 취급한다.

우선 분석 포인트:

- 어떤 웹 모듈이 공통 라이브러리에 의존하는가
- 공통 라이브러리 안에 어떤 service/dao 가 집중되는가
- 실제 온보딩 시작점이 웹 모듈보다 공통 라이브러리에 있어야 하는가

## 추천 시작점 규칙 초안

신규 입사자용 추천 시작점은 아래 기준으로 선정하는 편이 좋다.

- 공통 라이브러리의 핵심 service 패키지
- 실제 트래픽이 들어오는 web 모듈의 대표 controller
- 가장 많이 참조되는 config 파일
- 핵심 도메인의 DAO 와 mapper 연결 지점

## 낮은 신뢰도로 표시해야 하는 경우

- XML 참조가 깨져 있는 경우
- Java 코드와 mapper 연결이 이름 기반 추론에만 의존하는 경우
- 지역/국가별 분기 설정이 과도하게 많은 경우
- 리플렉션 또는 문자열 기반 동적 로딩이 많은 경우

## 산출 결과 예시

이 프로파일이 잘 동작하면 최소한 아래 같은 결과를 만들 수 있어야 한다.

- `legacy-web` 모듈은 `shared-lib` 에 의존한다.
- `web.xml` 은 특정 dispatcher servlet 을 설정한다.
- 해당 servlet 은 Spring XML 을 통해 controller bean 과 연결된다.
- 대표 controller 는 특정 service 를 호출한다.
- 해당 service 는 특정 DAO 를 통해 mapper statement 를 실행한다.
- 결과는 특정 JSP 로 렌더링된다.

## 품질 검증 포인트

이 프로파일의 분석 결과는 아래 관점에서 검증하는 편이 좋다.

- 대표 URL 하나를 선택했을 때 controller 까지 도달하는가
- 대표 controller 에서 service/dao 까지 이어지는가
- DAO 와 mapper/sql 이 연결되는가
- 주요 JSP 또는 view 이름이 연결되는가
- 공통 라이브러리 의존성이 누락되지 않는가

## 첫 구현의 범위 제한

첫 구현에서 아래는 과도하게 욕심내지 않는 편이 좋다.

- 완전한 메서드 수준 call graph
- 모든 환경 분기 정밀 해석
- 완전한 WebLogic/JEUS 동작 재현
- 모든 XML 변종 자동 지원

핵심은 `온보딩에 필요한 설명 가능한 구조`를 먼저 만드는 것이다.
