# 어댑터 인터페이스 초안

## 문서 목적

이 문서는 code2me의 범용 분석기 구조에서 `어댑터`가 어떤 형태로 정의되고 실행되어야 하는지에 대한 초안을 정리한 문서다.

목표는 아래를 명확히 하는 것이다.

- 어댑터의 책임 범위
- 어댑터가 받는 입력
- 어댑터가 내보내는 결과
- 어댑터 등록 방식
- 프로파일과 어댑터의 연결 방식
- 실행 순서와 의존성 처리 방식

## 어댑터의 역할

어댑터는 특정 파일 유형 또는 특정 기술 요소를 읽고, 그 안에서 구조 정보를 추출하는 실행 모듈이다.

어댑터는 아래까지만 책임진다.

- 입력 소스 탐색 또는 수집
- 파일 파싱
- 구조 정보 추출
- 공통 메타모델에 넣기 쉬운 중간 결과 생성
- 경고와 오류 보고

어댑터는 아래를 직접 책임지지 않는다.

- 제품 전체 해석 정책 결정
- 최종 온보딩 시작점 판단
- 여러 어댑터 결과의 전역 의미 부여
- 사용자 화면 구성

즉, 어댑터는 `부분 추출기`이고, 전체 의미는 프로파일과 분석 엔진이 조합해서 만든다.

## 설계 원칙

- 어댑터는 작고 명확한 책임을 가져야 한다.
- 어댑터는 특정 언어 전체보다 `특정 입력 유형` 중심으로 나누는 편이 좋다.
- 어댑터 출력은 공통 메타모델에 쉽게 매핑 가능해야 한다.
- 어댑터는 실패해도 전체 분석이 완전히 중단되지 않도록 설계해야 한다.
- 어댑터는 경고와 불확실성을 명시적으로 남겨야 한다.

## 어댑터 기본 구성

어댑터는 최소한 아래 정보를 가져야 한다.

- 고유 ID
- 이름
- 버전
- 설명
- 지원 입력 패턴
- 지원 대상 기술 태그
- 필요한 선행 어댑터
- 출력 가능한 엔터티 유형
- 출력 가능한 관계 유형

예:

- ID: `ant-build-xml`
- 이름: `Ant build.xml Adapter`
- 지원 입력 패턴: `**/build.xml`
- 기술 태그: `java`, `ant`, `legacy-java-ee`

## 어댑터 입력 모델 초안

어댑터는 분석 엔진으로부터 `AdapterContext`를 받는 구조가 적절하다.

`AdapterContext`는 아래 정보를 포함한다.

- 프로젝트 ID
- 프로젝트 루트 경로
- 현재 선택된 프로파일 ID
- 현재 파일 인덱스
- 어댑터 설정값
- 선행 어댑터의 결과 참조
- 실행 제한 조건

예시 항목:

- `projectId`
- `projectRoot`
- `profileId`
- `indexedFiles`
- `config`
- `upstreamResults`
- `executionLimits`

## 어댑터 출력 모델 초안

어댑터는 실행 결과로 `AdapterResult`를 반환하는 구조가 적절하다.

`AdapterResult`는 최소한 아래를 포함해야 한다.

- 생성한 노드 후보
- 생성한 관계 후보
- 발견한 엔트리포인트 후보
- 발견한 설정 정보
- 경고
- 오류
- 메트릭
- 디버깅용 추적 정보

예시 항목:

- `nodes`
- `edges`
- `entrypoints`
- `artifacts`
- `warnings`
- `errors`
- `metrics`
- `trace`

## 노드와 관계 출력 원칙

어댑터는 가능한 한 공통 메타모델의 후보를 만든다는 관점으로 출력해야 한다.

예:

- `web.xml` 어댑터
  - config 노드
  - servlet/filter/listener 관련 노드
  - `configures`, `routes_to` 성격의 관계 후보

- Java 소스 어댑터
  - class, interface, package, controller, service, dao 성격의 노드 후보
  - `contains`, `depends_on`, `calls` 관계 후보

- MyBatis 어댑터
  - mapper, sql statement 노드 후보
  - `queries`, `maps_to` 관계 후보

## 경고와 오류 모델

범용 분석기에서 중요한 것은 `정확도와 불확실성 표시`다.

따라서 어댑터는 실패를 단순 문자열로만 반환하면 안 되고, 구조화된 경고와 오류를 남기는 편이 좋다.

권장 필드:

- `code`
- `message`
- `severity`
- `filePath`
- `line`
- `recoverable`
- `details`

예:

- XML 파싱 실패
- 참조된 설정 파일 누락
- namespace 불일치
- 추론은 되었지만 확신이 낮은 연결

## 실행 상태 모델

어댑터 결과에는 실행 상태가 포함되는 편이 좋다.

권장 상태:

- `success`
- `partial_success`
- `skipped`
- `failed`

이 구조가 있으면 일부 어댑터 실패 시에도 전체 분석을 계속 진행할 수 있다.

## 어댑터 인터페이스 개념 초안

구현 언어와 무관하게 개념적으로는 아래 형태가 적절하다.

```ts
interface AnalyzerAdapter {
  id: string;
  name: string;
  version: string;
  capabilities: AdapterCapabilities;
  canRun(context: AdapterContext): boolean;
  collectInputs(context: AdapterContext): Promise<AdapterInputSet>;
  run(context: AdapterContext, inputs: AdapterInputSet): Promise<AdapterResult>;
}
```

핵심 포인트는 아래다.

- `canRun`: 현재 프로젝트와 프로파일에서 실행 가능한지 확인
- `collectInputs`: 필요한 파일 후보를 수집
- `run`: 실제 파싱과 추출 수행

## 등록 정보 모델 초안

어댑터는 레지스트리에 아래 정보로 등록되는 구조가 적절하다.

- `id`
- `name`
- `version`
- `supportedFilePatterns`
- `supportedProfiles`
- `technologyTags`
- `dependsOn`
- `produces`

예:

```json
{
  "id": "spring-xml",
  "name": "Spring XML Adapter",
  "version": "0.1.0",
  "supportedFilePatterns": ["**/applicationContext*.xml", "**/*dispatcher*.xml"],
  "supportedProfiles": ["legacy-java-ee", "spring-xml-webapp"],
  "technologyTags": ["java", "spring", "xml"],
  "dependsOn": ["web-xml"],
  "produces": ["config", "bean", "entrypoint_hint"]
}
```

## 프로파일과 어댑터의 연결 방식

프로파일은 직접 파일을 파싱하지 않고, 어떤 어댑터를 어떤 순서와 조건으로 사용할지 정의하는 편이 좋다.

즉, 프로파일은 아래를 가진다.

- 필요한 어댑터 목록
- 선택적 어댑터 목록
- 실행 순서 규칙
- 결과 해석 규칙
- 우선순위 규칙

예:

- `Legacy Java EE 프로파일`
  - 필수: `ant-build-xml`, `web-xml`, `spring-xml`, `java-source`
  - 선택: `jsp`, `sitemesh`, `mybatis`, `ibatis`

## 의존성과 실행 순서

어댑터는 서로 독립적일 수도 있지만, 일부는 선행 결과를 필요로 한다.

예:

- `spring-xml` 어댑터는 `web.xml` 어댑터 결과를 참조할 수 있다.
- `mybatis` 어댑터는 Java DAO 스캔 결과와 함께 쓰일 때 가치가 높다.

따라서 분석 엔진은 단순 리스트 실행이 아니라 `의존성 기반 실행 그래프`를 가질 필요가 있다.

권장 원칙:

- 선행 의존이 없으면 병렬 실행 가능
- 선행 결과가 필요한 경우 순차 실행
- 선택 어댑터 실패는 전체 실패로 보지 않음

## 어댑터 결과 병합 원칙

여러 어댑터가 같은 객체를 가리킬 수 있으므로 결과 병합 규칙이 필요하다.

예:

- Java 소스 어댑터가 `UserDao` 클래스를 찾음
- MyBatis 어댑터가 `userMapper.xml` 과 statement 를 찾음
- 프로파일 해석기가 둘을 연결해 `DAO -> SQL Mapper` 흐름을 형성

즉, 어댑터는 가능한 한 사실 기반 조각을 내고, 전역 병합은 코어에서 처리하는 편이 안전하다.

## 최소 인터페이스와 확장 인터페이스

처음부터 너무 무거운 인터페이스를 만들 필요는 없다.

초기에는 아래 정도의 최소 인터페이스면 충분하다.

- `id`
- `canRun`
- `collectInputs`
- `run`

이후 확장 가능 항목:

- 증분 분석 지원
- 캐시 키 계산
- 신뢰도 점수
- 추천 프로파일 기여도
- 디버그 리포트 생성

## 범용 분석기 관점에서 중요한 이유

이 인터페이스가 명확해야 아래가 가능해진다.

- 새 프로젝트 타입 추가
- 새 파일 포맷 추가
- AI 기반 어댑터 생성 자동화
- 부분 실패 허용
- 병렬 실행 최적화

즉, 어댑터 인터페이스는 단순 구현 상세가 아니라 범용성의 핵심 경계다.

## 현재 우선 적용 방향

현재는 `Legacy Java EE 프로파일`을 먼저 지원하되, 인터페이스 자체는 특정 스택에 종속되지 않게 잡는 것이 맞다.

따라서 첫 구현은 아래 순서가 적절하다.

1. 어댑터 공통 인터페이스 정의
2. 레지스트리 구조 정의
3. 실행 그래프 규칙 정의
4. Legacy Java EE용 최소 어댑터 구현
