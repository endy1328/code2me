# v1 제품 초안 및 언어 우선순위

## 문서 목적

이 문서는 현재까지 합의된 제품 방향을 한 곳에 정리한 기준 문서다.

- v1의 문제 정의와 범위
- 자동 분석과 수동 보강의 역할
- 협업 및 변경 이력 정책
- 첫 버전에서 우선 지원할 언어/프레임워크
- 향후 다른 언어 버전으로 확장할 때 유지할 기준

## 현재 기준 제품 한 줄 정의

신규 입사 개발자가 모노레포를 빠르게 이해할 수 있도록, 폴더/패키지와 파일 관계를 자동 분석하고 사람이 설명을 보강할 수 있게 만드는 온보딩용 코드 지식 도구를 만든다.

## 해결하려는 핵심 문제

첫 버전의 1순위 문제는 아래와 같다.

- 신규 입사 개발자가 모노레포의 구조와 진입점을 빠르게 파악하기 어렵다.

이 문제를 풀기 위해 v1은 대규모 범용 분석 플랫폼보다 "온보딩 시작점 제공"에 집중한다.

## v1 제품 초안

### 목적

- 신규 입사 개발자의 모노레포 온보딩

### 분석 범위

- 폴더/패키지 구조
- 파일 간 관계
- 관련 코드 찾기
- 온보딩 시작점 보기
- 패키지/폴더/파일 설명 보기

### 분석 방식

- 자동 분석이 기본이다.
- 사람이 수동으로 설명과 맥락을 보강할 수 있다.

### 메타데이터 저장

- 분석 결과와 설명 메타데이터는 프로그램 내부에 저장한다.

### 원본 소스 처리 원칙

- 원본 소스 코드는 절대 수정하지 않는다.

### 협업 정책

- 모든 개발자가 입력 가능하다.

### 품질 관리 및 변경 관리

- 승인 워크플로우는 v1에서 두지 않는다.
- 대신 수정 이력만 관리한다.
- 즉, 누가, 언제, 무엇을 바꿨는지는 남긴다.

이 정책은 첫 버전에서 입력 장벽을 낮추고 지식을 빠르게 쌓는 데 유리하다. 품질 문제가 생기면 승인 체계를 먼저 넣기보다 이력 기반으로 정리할 수 있다.

## 왜 이 범위가 적절한가

현재 정의된 범위는 아래 균형을 맞춘다.

- 자동 분석만으로 부족한 맥락은 사람이 채울 수 있다.
- 원본 코드 불변 원칙을 지켜 저장소 신뢰를 해치지 않는다.
- 협업 입력을 넓게 허용해 초반 지식 축적 속도를 확보한다.
- 승인 흐름 없이도 변경 이력을 남겨 책임 추적성과 복구 가능성을 확보한다.

## 자동 분석 우선 지원 언어/프레임워크

### 결론

v1의 1순위 지원 대상은 `TypeScript/JavaScript 중심 모노레포`로 정한다.

### 이유

- 프론트엔드 및 풀스택 모노레포에서 매우 흔하다.
- `import/export` 관계 분석이 상대적으로 명확하다.
- `package.json`, workspace 설정, `tsconfig` 등을 통해 패키지 구조와 진입점을 잡기 쉽다.
- 첫 버전 핵심 기능인 "관련 코드 찾기", "시작점 보기", "패키지/폴더/파일 설명"과 잘 맞는다.

### v1에서 우선 고려할 대상

- TypeScript
- JavaScript
- React 기반 패키지
- Next.js 애플리케이션
- Node.js 라이브러리/서비스 패키지
- `pnpm`, `yarn`, `npm workspaces` 기반 모노레포

### 우선 분석 후보

- `package.json` 기반 패키지 경계
- `import/export` 기반 파일 의존 관계
- 앱 엔트리포인트
- 라우트 또는 주요 시작 파일
- 설정 파일과 워크스페이스 연결 관계

## 이번 단계에서 보류한 선택지

### Java/Kotlin 중심

- 엔터프라이즈 환경 수요는 크지만 빌드 시스템과 구조 해석 난이도가 높다.

### Python 중심

- 시작은 가능하지만 동적 import와 실행 맥락 문제로 초반 분석 신뢰도 관리가 더 어렵다.

### 복수 언어 동시 지원

- 첫 버전 범위가 과하게 커지고 정확도 관리가 어려워진다.

## v1 범위 문장 제안

첫 버전은 `TypeScript/JavaScript 모노레포에서 패키지/폴더/파일 관계와 온보딩 시작점을 보여주는 도구`로 정의한다.

## v1 제외 범위 제안

아래 항목은 첫 버전에서 기본 제외 대상으로 본다.

- 복수 언어 동시 정밀 분석
- 승인 기반 협업 워크플로우
- 원본 코드 자동 수정
- 대규모 서비스 런타임 맵
- 고급 심볼 수준 정적 분석 전반

## 향후 다른 언어 버전으로 확장할 때 유지할 기준

나중에 Java/Kotlin, Python 등 다른 언어 버전으로 확장하더라도 아래 원칙은 그대로 유지하는 편이 좋다.

- 제품 목적은 온보딩 문제 해결에 고정한다.
- 원본 소스는 수정하지 않는다.
- 자동 분석과 수동 보강을 함께 쓴다.
- 협업은 폭넓게 허용하되 변경 이력은 반드시 남긴다.
- 언어 지원은 한 번에 하나씩 늘린다.
- 새 언어를 추가할 때도 패키지/폴더/파일 설명과 시작점 탐색 경험을 먼저 맞춘다.
