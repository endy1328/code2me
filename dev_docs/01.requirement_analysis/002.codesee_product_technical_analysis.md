# CodeSee 제품 및 기술 분석

## 문서 목적

이 문서는 CodeSee가 공개 자료상 어떤 입력을 받고, 어떤 산출물을 만들며, 어떤 워크플로우로 동작하는지 제품/기술 관점에서 정리한다. 일부 아키텍처 해석은 공개 문서를 바탕으로 한 추론이며, 그런 부분은 별도로 표기한다.

## 제품 기능 분해

### 1. Codebase Maps

Codebase Maps는 저장소 수준 코드 구조를 지도처럼 시각화하는 핵심 기능이다.

- 디렉터리, 파일, 연결 관계를 자동 생성된 맵으로 보여준다.
- 상류/하류 의존성 탐색이 가능하다고 설명한다.
- 최신 활동과 커밋이 많은 영역 같은 개발 문맥도 맵 관점에서 보려는 의도가 있다.
- 맵을 필터링해 특정 목적용 커스텀 뷰를 만들 수 있다.

제품 설계 관점에서 보면, 이 기능은 최소한 아래 능력을 전제로 한다.

- 저장소 수집
- 언어별 의존성 추출
- 그래프 생성
- 레이아웃 계산
- 맵 탐색 UI

### 2. Review Maps

Review Maps는 PR을 코드베이스 맥락 안에서 보여주는 기능이다.

- 저장소에 Maps가 설치되면 PR마다 자동으로 생성된다고 문서가 명시한다.
- 변경 파일뿐 아니라 비변경 파일과 연결 관계도 함께 보여 줄 수 있다.
- 작성자 기준으로 리뷰 순서를 안내하는 Tour를 붙일 수 있다.
- 리뷰어는 파일 나열보다 논리적 덩어리 단위로 검토할 수 있다.

비슷한 제품을 만들려면 핵심은 단순 diff 렌더링이 아니라 아래 조합이다.

- 변경 파일 집합 추출
- 변경 파일에서 연관 파일로 그래프 확장
- PR별 부분 그래프 생성
- 리뷰 맥락 저장
- 코드 호스트와의 링크 또는 주석 통합

### 3. 맵 편집과 협업

공식 문서는 맵 위에서 아래 작업이 가능하다고 설명한다.

- 라벨 부여
- 노트 작성
- 연결 파일 보기/숨기기
- 코드 열기
- 파일/폴더 단위 코멘트
- 실시간 협업
- View / Edit / Playground 모드

이 점이 중요하다. CodeSee는 자동 생성 맵만 제공하는 것이 아니라, 사람이 지식을 덧입히는 협업 캔버스로 확장되어 있다.

### 4. Service Maps

Service Maps는 코드 정적 분석만으로는 보이지 않는 런타임 구조를 다룬다.

- 공식 사이트는 서비스, 의존성, 데이터베이스, API, 흐름을 이해한다고 설명한다.
- 공식 문서는 Datadog, OpenTelemetry 연동 문서를 제공한다.
- Flows와 Variants를 통해 서비스 경로와 데이터 변형을 탐색하게 한다.
- "APM과 repo를 연결하면 10분 내 시작"이라는 메시지가 있다.

즉, CodeSee는 정적 코드 그래프만이 아니라 관측성 데이터와 저장소를 결합하는 방향으로 확장되었다.

### 5. Code Automations

공식 사이트는 Code Automation을 다음과 같이 설명한다.

- 변경을 감시하고 머지 전에 알림을 보낸다.
- 적절한 팀에 자동으로 문맥 포함 알림을 보낸다.
- 베스트 프랙티스, 보안, gotcha를 강제 적용한다.

문서/과금 페이지를 보면 Business 티어에서 트리거 수 제한이 존재한다. 이 기능은 단순 알림이 아니라 "코드 그래프 기반 규칙 엔진" 성격에 가깝다.

### 6. AI

공식 문서 기준으로 CodeSee AI는 다음을 제공한다.

- 파일 단위 AI 요약
- Review Map에서 변경 파일별 요약
- AI가 Tour 초안을 만드는 지원
- 코드/서비스 이해를 위한 요약 생성

기술적으로 중요한 사실은 다음이다.

- Anthropic Claude API를 사용한다고 문서가 명시한다.
- "small amounts of your code or code changes"가 Anthropic으로 전송된다고 설명한다.
- 해당 데이터는 모델 학습에 사용되지 않는다고 문서가 안내한다.
- AI 기능은 Enterprise 전용이라고 안내된 부분이 있다.

## 입력 데이터 관점 분석

CodeSee가 공개적으로 다루는 입력 소스는 아래와 같이 정리된다.

- Git 저장소
- Pull Request / code review metadata
- 언어별 소스코드 구조
- 저장소 설정 파일
- Datadog APM 데이터
- OpenTelemetry 데이터
- 사용자 수동 주석 데이터

이 입력 구조는 "정적 코드 이해 + 런타임 컨텍스트 + 협업 문맥"의 조합이다.

## 출력 데이터 관점 분석

CodeSee의 주요 출력물은 아래와 같다.

- 전체 코드베이스 맵
- PR 영향 맵
- 서비스 맵
- 데이터 흐름 뷰
- 라벨/노트/코멘트가 포함된 지식 맵
- AI 요약, AI 투어, 자동화 알림

즉 출력물이 단순 다이어그램이 아니라 운영 가능한 지식 레이어까지 포함한다.

## 설정과 정확도

공식 문서에 따르면 CodeSee는 자동 감지를 우선하지만, 정확도가 부족하면 `.codesee.json`으로 보정한다.

- Python monorepo의 경우 프로젝트 목록과 `pythonpath` 정보를 줄 수 있다.
- 일부 프로젝트는 `setup.py`, `setup.cfg`, `pyproject.toml` 유무에 따라 자동 인식 범위가 달라진다.

이 지점은 제품적으로 중요하다. "자동 생성"을 전면에 내세우더라도 실제 대형 코드베이스에서는 언어/빌드/모노레포 구조 때문에 수동 힌트가 필요하다.

## 권한 및 배포 모델에 대한 추정

공개 자료를 종합하면 CodeSee는 대체로 다음 배포 모델을 가진 것으로 보인다.

- 저장소 연동형 SaaS
- 코드 호스트와 연동되는 앱 또는 설치 흐름
- 런타임 맵을 위해서는 APM/OTel 연동 필요
- UI는 웹 기반 협업 캔버스를 중심으로 제공
- 일부 경험은 VS Code 확장으로 편입

이 부분은 공개 페이지를 바탕으로 한 추정이다. 하지만 제품 설명, 문서, 과금 구조와 잘 부합한다.

## 기술 아키텍처 추정

공개 자료를 바탕으로 비슷한 제품을 구현할 때 필요한 기술 블록을 역산하면 아래와 같다.

1. Repository ingestion layer
2. Language analyzers / dependency extractors
3. Graph builder and graph store
4. Incremental update pipeline for PRs and repo changes
5. Runtime trace ingestion for service maps
6. Layout/rendering engine
7. Collaboration metadata store for labels, notes, comments, tours
8. Rule engine for automations
9. LLM orchestration layer for summaries/tours
10. Host integrations with Git providers and IDEs

## 제품 설계상 인상적인 지점

- 그래프를 여러 워크플로우에 재사용한다. 같은 기초 그래프가 코드 이해, 리뷰, 서비스 분석, 자동화, AI에 모두 쓰인다.
- 자동 생성 그래프에 인간의 팀 지식을 얹는다. 이것이 단순 시각화 툴과의 차이점이다.
- 정적 구조와 런타임 구조를 모두 다루려 한다. 이게 제품 상한을 높인다.
- PR 리뷰를 별도 제품이 아니라 맵의 부분 뷰로 해석한다. 데이터 재사용성이 높다.

## 제품 설계상 어려운 지점

- 대규모 코드베이스에서 그래프 생성/레이아웃/렌더링 성능
- 언어별 분석 정확도와 빌드 시스템 다양성 대응
- 잘못된 연결 관계가 사용자 신뢰를 크게 훼손할 위험
- 런타임 데이터와 정적 코드 노드 연결의 난이도
- 멀티리포와 모노레포를 모두 만족시키는 UX 설계
- 협업 메타데이터와 자동 재생성 간 충돌 처리

## 비슷한 제품 개발 시 해석

- CodeSee의 본체는 "예쁜 그래프"가 아니라 "지속적으로 갱신되는 코드 지식 그래프 플랫폼"에 가깝다.
- MVP를 그래프 렌더링에서 시작할 수는 있지만, 실제 차별화는 `영향 분석`, `리뷰 맥락화`, `협업 주석`, `신뢰 가능한 자동 갱신`에서 나온다.
- 서비스 맵과 AI는 매력적이지만, 초기 제품에서는 코어 그래프와 리뷰/온보딩 흐름이 먼저다.
