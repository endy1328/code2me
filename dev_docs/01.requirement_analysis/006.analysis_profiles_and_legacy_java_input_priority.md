# 분석 프로파일과 Legacy Java 입력 소스 우선순위

## 문서 목적

이 문서는 제품을 하나의 고정 언어 도구로 보지 않고, 프로젝트 타입별로 다른 분석 규칙을 적용하는 구조로 정리하기 위한 기준 문서다.

특히 현재 먼저 처리해야 하는 전통적 엔터프라이즈 Java 시스템을 기준으로 아래를 명확히 한다.

- 분석 프로파일 개념
- 현재 프로젝트에 해당하는 프로파일 정의
- 이 프로파일에서 우선 읽어야 하는 입력 소스
- 진입점 탐색 기준
- persistence 계층 해석 원칙

## 핵심 정리

제품 목표는 여전히 `다용도 분석 툴`이다.

다만 실제 분석은 모든 프로젝트를 같은 방식으로 해석하지 않고, `프로젝트 타입별 분석 프로파일`에 따라 다른 입력 소스와 규칙을 적용한다.

예시:

- TS/JS 모노레포 프로파일
- Legacy Java EE 프로파일
- Python 애플리케이션 프로파일

즉, 툴은 하나이고 분석 규칙 세트가 여러 개 있는 구조로 간다.

## 현재 먼저 처리할 프로젝트 타입

현재 우선 대응 대상은 아래와 같은 전통적 엔터프라이즈 Java 시스템이다.

- Java 8
- Ant `build.xml` 기반 빌드
- EAR/WAR 배포 구조
- Spring MVC
- XML 기반 `applicationContext`
- `web.xml` 중심 설정
- JSP + SiteMesh
- Servlet 2.5 스타일 설정
- iBATIS 2.x 와 MyBatis 3.x 혼합
- SQL/XML mapper 중심 데이터 접근
- JEUS/WebLogic 계열 전통 WAS 운영환경
- 공통 라이브러리 위에 여러 웹/배치 모듈이 얹힌 구조

## 현재 프로젝트에 적용할 분석 프로파일

이 프로젝트는 `Legacy Java EE 프로파일`로 정의한다.

이 프로파일의 핵심 원칙은 아래와 같다.

- 최신 Spring Boot 애플리케이션으로 가정하지 않는다.
- annotation 만으로 진입점을 찾지 않는다.
- XML 설정과 배포 구조를 우선 해석한다.
- 공통 라이브러리와 웹 모듈의 역할을 분리해서 본다.
- Java 코드와 XML mapper 를 함께 읽어 persistence 흐름을 본다.

## 분석 시 가장 중요한 해석 원칙

### 1. Spring Boot 기준 추론 금지

- `@SpringBootApplication`
- embedded server
- convention based auto-configuration

위와 같은 최신 Spring Boot 전제를 두고 구조를 추론하면 실제 시스템을 오해할 가능성이 높다.

### 2. 진입점은 XML + web 계층에서 먼저 찾는다

우선 진입점 후보는 아래 순서로 본다.

1. 각 모듈의 `web.xml`
2. `dispatcherServlet.xml` 또는 이에 준하는 servlet context 설정 파일
3. `applicationContext*.xml`
4. 이후 `@Controller` 클래스

즉, annotation 은 보조 신호이고 XML 설정이 기준이다.

### 3. shared-lib 를 중심축으로 본다

비즈니스 로직의 상당수가 `shared-lib`에 있고 웹 모듈은 채널별 프레젠테이션과 컨트롤러 역할이 강하므로, 웹 모듈만 보고 시스템을 해석하면 실제 구조를 놓친다.

따라서 분석 결과는 아래를 분리해서 보여줄 수 있어야 한다.

- 공통 비즈니스 로직 계층
- 채널별 웹 진입 계층
- 배치 또는 백그라운드 처리 계층

### 4. DAO 와 SQL mapper 를 한 흐름으로 본다

DAO 구현체만 보면 실제 쿼리를 놓치고, XML mapper 만 보면 호출 주체를 놓친다.

따라서 아래를 연결된 흐름으로 취급해야 한다.

- Controller / Service
- DAO / Repository 성격의 Java 클래스
- iBATIS `sql-map.xml`
- MyBatis `mapper.xml`
- SQL statement id

### 5. iBATIS 와 MyBatis 를 같은 persistence 계층으로 정규화한다

기술적으로는 세대가 다르지만, 온보딩 관점에서는 둘 다 `Java 코드와 연결된 XML 기반 SQL 정의`라는 공통 역할을 가진다.

따라서 제품에서는 세부 구현 차이를 숨기지 않되, 상위 개념상 하나의 persistence 레이어로 묶어 다룰 수 있어야 한다.

## Legacy Java 프로파일의 분석 입력 소스 우선순위

아래는 첫 분석에서 우선 읽어야 하는 입력 소스 순서다.

### P0. 구조와 진입점 파악용

- `build.xml`
- 모듈별 `web.xml`
- `dispatcherServlet.xml` 또는 servlet context XML
- `applicationContext*.xml`

이 단계의 목적은 모듈 경계, 배포 단위, 스프링 빈 구성, 웹 진입 흐름을 파악하는 것이다.

### P1. 실제 코드 흐름 파악용

- `*.java`
- `@Controller`, `@Service`, `@Repository` 또는 유사 역할 클래스
- 공통 라이브러리 모듈 소스
- 배치 실행 관련 Java 클래스

이 단계의 목적은 진입점에서 공통 라이브러리와 서비스 계층으로 이어지는 호출 구조를 연결하는 것이다.

### P2. 데이터 접근 해석용

- iBATIS `sql-map*.xml`
- MyBatis `*mapper.xml`
- DAO 관련 Java 클래스
- SQL statement namespace / id 참조

이 단계의 목적은 Java 메서드와 실제 SQL 정의를 연결하는 것이다.

### P3. 화면 및 프레젠테이션 해석용

- `*.jsp`
- SiteMesh 설정 파일
- view resolver 관련 XML 설정
- 레이아웃/데코레이터 설정

이 단계의 목적은 컨트롤러에서 어떤 화면으로 이어지는지와 공통 레이아웃 구조를 파악하는 것이다.

### P4. 배포 및 운영환경 보강용

- EAR/WAR 관련 설정
- JEUS 관련 설정 흔적
- WebLogic 관련 설정 흔적
- 지역/국가별 설정 파일
- 환경별 properties / xml 설정

이 단계의 목적은 실제 운영 배치 형태와 멀티리전 분기 구조를 설명 메타데이터로 보강하는 것이다.

## 우선 수집 대상 파일 패턴 제안

Legacy Java 프로파일에서 초기에 수집 우선순위를 높게 둘 파일 패턴은 아래와 같다.

- `**/build.xml`
- `**/web.xml`
- `**/*dispatcher*.xml`
- `**/applicationContext*.xml`
- `**/*.java`
- `**/*sql-map*.xml`
- `**/*mapper.xml`
- `**/*.jsp`
- `**/*sitemesh*.xml`
- `**/*decorator*.xml`
- `**/*.properties`

## 진입점 탐색 규칙 초안

진입점 후보는 아래 순서로 만든다.

1. `web.xml` 에 선언된 servlet, filter, listener 확인
2. dispatcher servlet 이 참조하는 XML context 확인
3. XML 에 등록된 controller, handler mapping, view resolver 확인
4. 이후 annotation 기반 `@Controller` 와 request mapping 확인
5. JSP / SiteMesh 연결 확인

즉, URL 진입은 코드 annotation 하나로 찾는 것이 아니라 `배포 서술자 -> Spring XML -> Controller -> View` 순서로 따라간다.

## 공통 라이브러리 해석 규칙 초안

`shared-lib` 같은 공통 라이브러리는 단순 하위 모듈이 아니라 시스템 중심 지식 저장소로 취급해야 한다.

초기 분석에서는 아래를 우선 정리하는 편이 좋다.

- 어떤 웹 모듈이 공통 라이브러리를 참조하는가
- 공통 라이브러리 안에 어떤 비즈니스 도메인이 있는가
- 웹 모듈이 실제로는 어느 서비스/DAO 를 호출하는가

## persistence 계층 정규화 원칙

제품 내부 모델에서는 아래처럼 통합하는 편이 좋다.

- 상위 개념: `Persistence Mapping`
- 하위 타입:
  - `iBATIS SqlMap`
  - `MyBatis Mapper`
  - `DAO Method`
  - `SQL Statement`

이렇게 하면 기술 차이를 보존하면서도 온보딩 화면에서는 일관된 탐색 경험을 줄 수 있다.

## 이 문서가 의미하는 제품 방향

이 문서는 TS/JS 중심 결정과 충돌하지 않는다.

정리하면 아래처럼 이해하면 된다.

- 제품 장기 방향: 다용도 분석 툴
- 제품 구조 원칙: 프로젝트 타입별 분석 프로파일 적용
- 현재 실제 우선 대상: Legacy Java EE 프로파일
- 추후 추가 대상: TS/JS, Python 등 다른 프로파일

## 다음 설계 작업 제안

이 문서 다음으로 정리할 항목은 아래가 적절하다.

1. Legacy Java 프로파일용 내부 메타데이터 스키마 초안
2. `web.xml -> Spring XML -> Controller -> Service -> DAO -> Mapper` 연결 규칙 초안
3. shared-lib 중심 온보딩 시작점 선정 규칙
4. iBATIS/MyBatis 혼합 프로젝트용 관계 그래프 표현 방식
