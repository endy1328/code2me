# 범용 분석기를 위한 프로파일-어댑터 모델

## 문서 목적

이 문서는 code2me를 특정 언어 전용 도구가 아니라 `범용 분석기`로 개발하기 위한 구조 원칙을 정리한 기준 문서다.

특히 아래 용어를 헷갈리지 않도록 분리해서 정의한다.

- 입력 소스
- 분석 프로파일
- 어댑터
- 공통 메타모델
- 분석 엔진

## 제품의 최종 목적

code2me의 최종 목적은 특정 기술 스택 하나만 이해하는 도구가 아니라, 여러 종류의 프로젝트를 분석할 수 있는 `범용 분석기`를 만드는 것이다.

다만 범용 분석기는 처음부터 모든 언어를 동시에 정밀 지원하는 방식으로 만들기보다, 공통 구조를 먼저 만들고 프로젝트 타입별 규칙을 확장하는 방식으로 개발해야 한다.

즉, 제품은 아래 방향으로 가야 한다.

- 코어는 범용 구조를 가진다.
- 프로젝트별 차이는 프로파일과 어댑터로 분리한다.
- 분석 결과는 공통 메타모델에 모은다.

## 핵심 구조 한 줄 요약

- 입력 소스는 사용자가 제공한다.
- 프로파일은 어떻게 해석할지 결정한다.
- 어댑터는 실제로 읽고 추출한다.
- 분석 엔진은 프로파일과 어댑터를 조합해 실행한다.
- 결과는 공통 메타모델에 저장한다.

## 용어 정리

### 1. 입력 소스

입력 소스는 사용자가 분석 대상으로 제공하는 실제 프로젝트 파일과 디렉터리다.

예:

- `build.xml`
- `web.xml`
- `applicationContext.xml`
- `*.java`
- `package.json`
- `tsconfig.json`
- `*.jsp`
- `mapper.xml`

즉, 입력 소스는 분석 대상 그 자체다.

### 2. 분석 프로파일

분석 프로파일은 특정 프로젝트 타입을 어떻게 이해할지 정의한 규칙 세트다.

프로파일은 보통 아래를 가진다.

- 어떤 입력 소스를 우선 읽을지
- 어떤 어댑터를 사용할지
- 진입점을 어디서 찾을지
- 어떤 관계를 중요하게 추출할지
- 어떤 결과를 우선 보여줄지
- 어떤 오해를 피해야 할지

즉, 프로파일은 `해석 정책`이다.

예:

- `Legacy Java EE 프로파일`
- `Spring Boot 프로파일`
- `TS/JS Monorepo 프로파일`

### 3. 어댑터

어댑터는 특정 종류의 입력 소스를 실제로 읽고 구조 정보를 추출하는 실행 모듈이다.

예:

- Ant `build.xml` 어댑터
- `web.xml` 어댑터
- Spring XML 어댑터
- Java 소스 어댑터
- iBATIS/MyBatis XML 어댑터
- `package.json` 어댑터
- TypeScript import 어댑터

즉, 어댑터는 `해석기` 또는 `추출기`에 가깝다.

중요한 점은 `build.xml 어댑터`가 `build.xml 파일을 제공하는 것`이 아니라, `프로젝트 안의 build.xml을 읽어서 구조 정보를 추출하는 모듈`이라는 점이다.

### 4. 공통 메타모델

공통 메타모델은 서로 다른 프로파일과 어댑터가 만든 결과를 같은 형태로 담기 위한 내부 데이터 모델이다.

예:

- 노드: project, module, package, folder, file, class, controller, service, dao, mapper, sql, view, config
- 관계: contains, depends_on, imports, configures, routes_to, calls, queries, renders, deploys

범용 분석기를 만들려면 이 공통 메타모델이 코어가 되어야 한다.

### 5. 분석 엔진

분석 엔진은 선택된 프로파일을 보고 필요한 어댑터를 실행하고, 그 결과를 공통 메타모델로 합치는 오케스트레이션 계층이다.

즉, 분석 엔진은 아래 역할을 한다.

- 프로파일 로드
- 필요한 어댑터 선택
- 입력 소스 수집
- 어댑터 실행 순서 제어
- 결과 병합
- 최종 분석 결과 생성

## 관계 정리

헷갈리지 않게 한 줄씩 정리하면 아래와 같다.

- `입력 소스`: 분석할 실제 파일
- `프로파일`: 어떻게 해석할지 정하는 규칙
- `어댑터`: 파일을 읽고 정보 추출하는 모듈
- `분석 엔진`: 프로파일과 어댑터를 실행하는 조정자
- `공통 메타모델`: 결과를 저장하는 공통 구조

## 동작 흐름

범용 분석기의 기본 실행 흐름은 아래와 같다.

1. 사용자가 프로젝트 소스를 등록한다.
2. 시스템이 프로파일을 추천하거나 사용자가 직접 선택한다.
3. 선택된 프로파일이 필요한 입력 소스와 어댑터 목록을 결정한다.
4. 분석 엔진이 어댑터를 순서대로 실행한다.
5. 어댑터가 추출한 결과를 공통 메타모델에 적재한다.
6. 프로파일 규칙에 맞춰 시작점, 관계, 설명 후보를 만든다.
7. 사용자에게 프로파일에 맞는 분석 결과를 보여준다.

## 예시 1. Legacy Java EE 프로젝트

### 입력 소스

- `dispatcherServlet.xml`
- `applicationContext*.xml`
- `*.java`
- `sql-map.xml`
- `mapper.xml`
- `*.jsp`

### 프로파일

- `Legacy Java EE 프로파일`

### 어댑터

- Ant `build.xml` 어댑터
- `web.xml` 어댑터
- Spring XML 어댑터
- Java 소스 어댑터

- JSP/SiteMesh 어댑터

### 결과

- 모듈 구조
- 배포 단위
- `web.xml -> Spring XML -> Controller -> Service -> DAO -> Mapper -> SQL` 흐름
- shared-lib 중심 비즈니스 구조

- 앱/라이브러리 연결 구조

## 왜 이 구조가 범용 분석기에 맞는가

이 구조를 쓰면 제품 코어를 흔들지 않고 새 프로젝트 타입을 추가할 수 있다.

- 새로운 프로젝트 타입이 와도 새 프로파일을 추가하면 된다.
- 기존 어댑터를 재사용할 수 있다.
- 새 파일 형식만 생기면 어댑터만 추가하면 된다.
- 결과는 공통 메타모델에 모이므로 UI와 저장 구조를 재사용할 수 있다.

즉, 범용성은 `모든 걸 한 번에 지원`해서 얻는 것이 아니라, `공통 코어 + 확장 가능한 프로파일/어댑터 구조`로 얻는다.

## 개발 원칙

범용 분석기라는 목표에 맞추려면 아래 원칙을 지켜야 한다.

- 제품 코어는 특정 언어 가정에 묶이면 안 된다.
- 프로파일은 추가 가능하고 교체 가능해야 한다.
- 어댑터는 작은 단위로 분리해야 한다.
- 결과 저장 구조는 공통 메타모델로 통합해야 한다.
- 특정 프로젝트 대응 로직은 코어에 하드코딩하지 말고 프로파일 또는 어댑터에 둬야 한다.

## 현재 우선 개발 방향

현재는 범용 분석기 코어를 염두에 두되, 실제 첫 적용 대상은 `Legacy Java EE 프로파일`로 본다.

즉, 개발 방향은 아래처럼 가져가는 편이 맞다.

- 1단계: 공통 메타모델과 프로파일/어댑터 실행 구조 설계
- 2단계: Legacy Java EE 프로파일과 필요한 어댑터 구현
- 3단계: 같은 코어 위에 TS/JS 프로파일 추가
- 4단계: 프로파일 자동 추천과 확장성 강화

## 권장 정리 문장

현재 기준으로 제품을 아래처럼 설명할 수 있다.

`code2me는 범용 코드 분석기를 목표로 하며, 프로젝트 타입별 분석 프로파일과 파일 유형별 어댑터를 조합해 소스를 해석하고 공통 메타모델에 구조 지식을 축적하는 도구다.`
