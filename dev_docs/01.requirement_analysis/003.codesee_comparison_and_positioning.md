# CodeSee 비교 관점 및 포지셔닝 분석

## 문서 목적

이 문서는 CodeSee와 유사하거나 인접한 도구 범주를 비교해, 어떤 차별점을 제품 요구사항으로 가져가야 하는지 정리한다. 여기서 핵심은 "어떤 경쟁 제품이 더 좋다"가 아니라 "CodeSee가 어느 지점에 서 있는 제품인가"를 분해하는 것이다.

## 먼저 결론

CodeSee는 아래 네 범주가 겹치는 지점에 있는 제품이다.

- 코드 검색/이해 플랫폼
- 시각적 코드 구조 맵
- 지속적 코드 문서화/온보딩 도구
- 리뷰/변경 영향 분석 도구

이 네 범주를 동시에 건드리기 때문에 설명은 어렵지만, 반대로 잘 구현되면 강한 제품 방어력이 생긴다.

## 비교 축

비슷한 프로그램을 설계할 때 가장 중요한 비교 축은 아래와 같다.

- 사용자 문제 정의: 온보딩, 리뷰, 리팩터링, 아키텍처 파악 중 무엇을 1순위로 푸는가
- 대상 사용자: 개인 개발자 중심인지 팀 협업 중심인지
- 입력 소스 범위: Git만 다루는지, PR/APM/OTel까지 연결하는지
- 시각화 단위: 파일, 함수, 모듈, 서비스, 호출 흐름 중 무엇이 핵심인지
- 갱신 방식: 배치 생성인지, 이벤트 기반 증분 갱신인지
- 협업 레이어: 코멘트, 노트, 투어, 공유 뷰 같은 팀 지식 관리가 있는지
- 인사이트 수준: 단순 맵인지, 영향 분석/알림/추천까지 제공하는지
- 온보딩 비용: 설치 후 첫 가치까지 걸리는 시간
- 확장성: 대규모 저장소와 멀티리포에서 유지되는지
- 신뢰성: 맵의 정확도와 최신성이 반복 사용을 견딜 정도인지

## 인접 제품군과 비교

### 1. Sourcegraph

공식 메시지 기준으로 Sourcegraph는 대규모 코드 검색, 코드 이해, 코드 인텔리전스, 대규모 변경 자동화에 강하다.

- 강점 축: 검색, 참조 탐색, 대규모 코드 변경, 에이전트/AI 문맥 제공
- 차이점: CodeSee는 "맵 중심 시각 인터페이스"가 더 핵심이고, Sourcegraph는 "검색과 코드 인텔리전스"가 더 중심이다
- 시사점: 유사 제품을 만든다면 검색형 제품이 될지, 맵형 제품이 될지 먼저 정해야 한다

### 2. Swimm

공식 문서 기준으로 Swimm은 코드와 결합된 지속적 문서화에 집중한다.

- 강점 축: 지속적 문서화, 코드-문서 동기화, 팀 지식 전파
- 차이점: CodeSee는 문서를 맵 위에서 시각 탐색 경험으로 풀고, Swimm은 문서 자체의 유지와 탐색에 더 가깝다
- 시사점: CodeSee와 비슷한 체감을 원하면 "문서 도구"보다는 "시각 탐색 도구 + 문서 레이어"가 되어야 한다

### 3. CodeScene

공식 자료 기준으로 CodeScene은 행동 기반 코드 분석, 핫스팟, 기술 부채 우선순위, 지식 사일로 탐지에 강하다.

- 강점 축: Git 이력 기반 위험도 분석, 조직/코드 상호작용 분석, 기술 부채 우선순위
- 차이점: CodeSee는 실시간 탐색형 맵과 리뷰 맥락이 중심이고, CodeScene은 운영/관리 관점의 분석과 우선순위화가 더 중심이다
- 시사점: 향후 확장 방향으로는 `CodeSee형 시각화`와 `CodeScene형 위험 분석`의 결합 여지가 크다

### 4. Structurizr

공식 문서 기준으로 Structurizr은 C4 모델 기반의 구조화된 아키텍처 다이어그램 도구다.

- 강점 축: 명시적 모델링, 아키텍처 문서화, 다이어그램 거버넌스
- 차이점: CodeSee는 코드에서 출발해 자동으로 지도를 생성하려 하고, Structurizr은 사람이 의도적으로 모델을 정의한다
- 시사점: 자동 생성 제품을 만들더라도, 팀이 의도한 추상화 레벨을 수동으로 고정하는 기능이 필요할 수 있다

## CodeSee 포지셔닝 요약

CodeSee는 아래와 같은 독특한 조합을 가진다.

- Sourcegraph처럼 코드 이해를 노린다.
- Swimm처럼 지식의 지속적 축적을 노린다.
- CodeScene처럼 리스크와 변경 맥락의 문제를 건드린다.
- Structurizr처럼 구조를 시각적으로 보여주려 한다.

하지만 구현 방식은 다르다. CodeSee는 이 모든 것을 "자동 생성되는 맵"을 중심 UI로 묶는다.

## 따라서 요구사항으로 가져가야 할 핵심

- 단순 의존성 그래프만으로는 부족하다.
- 사용자에게 "어디부터 봐야 하는지"를 안내하는 워크플로우가 있어야 한다.
- 공유 가능한 지식 레이어가 있어야 한다.
- 코드 변경과 구조 이해가 끊기지 않아야 한다.
- 정확하지 않은 맵은 오히려 해가 되므로 신뢰도 관리가 핵심이다.

## 추천 포지셔닝 옵션

비슷한 프로그램을 만들 때는 아래 셋 중 하나를 먼저 고르는 편이 좋다.

### 옵션 A. Review-first

PR 영향 분석과 리뷰 투어를 핵심으로 시작한다.

- 장점: 가치 전달이 빠르다
- 장점: 데이터 범위가 비교적 작다
- 한계: 전체 코드베이스 이해 제품으로 확장하려면 추가 투자가 필요하다

### 옵션 B. Onboarding-first

코드베이스 맵과 주석/노트/가이드 뷰를 핵심으로 시작한다.

- 장점: 레거시 코드 이해 문제에 직접 대응한다
- 장점: 팀 지식 축적 효과가 크다
- 한계: 초기에는 리뷰/자동화 같은 즉시 ROI가 약할 수 있다

### 옵션 C. Platform-first

코드 그래프 플랫폼을 먼저 만들고 리뷰, 서비스, AI, 자동화를 얹는다.

- 장점: 장기적으로 가장 강한 구조다
- 장점: 재사용성과 확장성이 높다
- 한계: 초기 개발 비용과 복잡도가 가장 높다

## 메인 추천

CodeSee와 닮은 제품을 현실적으로 만들려면 `Review-first` 또는 `Onboarding-first`로 시장 진입한 뒤, 내부 구현은 `Platform-first`를 염두에 두는 방식이 가장 실용적이다.
