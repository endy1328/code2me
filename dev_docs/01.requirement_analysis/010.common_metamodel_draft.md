# 공통 메타모델 초안

## 문서 목적

이 문서는 code2me를 범용 분석기로 개발하기 위해 필요한 `공통 메타모델`의 초안을 정리한 문서다.

목표는 아래를 명확히 하는 것이다.

- 서로 다른 프로파일과 어댑터 결과를 어떤 공통 구조에 담을 것인가
- 제품 코어가 어떤 엔터티와 관계를 기준으로 동작할 것인가
- 온보딩, 탐색, 설명, 변경 이력 기능이 어떤 모델 위에 올라갈 것인가

## 왜 공통 메타모델이 필요한가

프로파일과 어댑터는 프로젝트 타입마다 달라질 수 있다.

예:

- Legacy Java EE는 `web.xml`, Spring XML, DAO, mapper 를 본다.
- TS/JS Monorepo는 `package.json`, import/export, 라우트 파일을 본다.

하지만 제품 코어는 프로젝트마다 완전히 다른 저장 구조를 가지면 안 된다.

따라서 code2me는 분석 결과를 하나의 공통 구조로 저장해야 한다.

즉, `프로파일은 다를 수 있지만 결과 저장 구조는 최대한 공통이어야 한다`.

## 메타모델 설계 원칙

- 특정 언어 전용 타입에 코어를 묶지 않는다.
- 온보딩과 탐색에 필요한 의미 단위를 우선 모델링한다.
- 어댑터가 사실 기반 결과를 쉽게 넣을 수 있어야 한다.
- 프로파일별 특수성은 확장 필드로 흡수한다.
- 변경 이력과 수동 보강을 기본 모델에 포함한다.

## 메타모델의 최상위 구성

공통 메타모델은 아래 다섯 축으로 보는 것이 적절하다.

- 엔터티
- 관계
- 진입점
- 설명/주석
- 변경 이력

## 1. 엔터티 모델

엔터티는 분석 결과에서 식별 가능한 객체다.

최소 공통 엔터티 유형은 아래가 적절하다.

- `project`
- `module`
- `package`
- `folder`
- `file`
- `config`
- `class`
- `interface`
- `function`
- `controller`
- `service`
- `repository`
- `dao`
- `mapper`
- `sql_statement`
- `view`
- `route`
- `build_target`
- `deployment_unit`
- `batch_job`
- `external_system`

모든 프로젝트가 이 타입을 다 쓰는 것은 아니지만, 범용 분석기 코어는 이 정도 폭을 수용할 수 있어야 한다.

## 엔터티 공통 필드

모든 엔터티는 최소한 아래 필드를 가지는 편이 좋다.

- `id`
- `type`
- `name`
- `displayName`
- `projectId`
- `path`
- `language`
- `profileHints`
- `sourceAdapterIds`
- `confidence`
- `tags`
- `metadata`

설명:

- `id`: 전역적으로 안정적인 식별자
- `type`: 엔터티 유형
- `name`: 내부 식별용 이름
- `displayName`: UI 표시 이름
- `projectId`: 소속 프로젝트
- `path`: 파일 또는 논리 경로
- `language`: 관련 언어
- `profileHints`: 어떤 프로파일 해석과 관련되는지
- `sourceAdapterIds`: 어떤 어댑터가 이 엔터티를 만들었는지
- `confidence`: 추론 신뢰도
- `tags`: 추가 분류
- `metadata`: 프로파일 특화 속성

## 엔터티 계층 예시

예를 들어 Legacy Java EE에서는 아래처럼 연결될 수 있다.

- `project`
- `module`
- `folder`
- `file`
- `config`
- `controller`
- `service`
- `dao`
- `mapper`
- `sql_statement`
- `view`

TS/JS Monorepo에서는 아래처럼 연결될 수 있다.

- `project`
- `module`
- `package`
- `folder`
- `file`
- `route`
- `function`
- `config`

즉, 공통 메타모델은 모든 프로젝트가 같은 엔터티를 쓰게 하는 것이 아니라, `같은 틀에서 필요한 엔터티만 선택하게 하는 구조`여야 한다.

## 2. 관계 모델

관계는 엔터티 사이의 연결을 표현한다.

최소 공통 관계 유형은 아래가 적절하다.

- `contains`
- `depends_on`
- `imports`
- `configures`
- `declares`
- `routes_to`
- `calls`
- `queries`
- `renders`
- `maps_to`
- `implements`
- `extends`
- `deploys`
- `uses`
- `references`

## 관계 공통 필드

모든 관계는 최소한 아래 필드를 가지는 편이 좋다.

- `id`
- `type`
- `from`
- `to`
- `projectId`
- `sourceAdapterIds`
- `confidence`
- `directional`
- `metadata`

설명:

- `from`, `to`: 관계 양 끝 엔터티 ID
- `directional`: 방향성 존재 여부
- `metadata`: 라인 번호, statement id, bean name 같은 세부 정보

## 관계 모델 예시

Legacy Java EE:

- `module contains file`
- `web.xml configures dispatcherServlet`
- `controller calls service`
- `service calls dao`
- `dao queries sql_statement`
- `controller renders view`

TS/JS:

- `package contains file`
- `file imports file`
- `route routes_to file`
- `module depends_on module`

## 3. 진입점 모델

온보딩 도구에서 중요한 것은 단순 구조보다 `어디서 시작해야 하는가`다.

따라서 진입점은 별도 모델로 두는 편이 좋다.

권장 진입점 유형:

- `web_entry`
- `api_entry`
- `batch_entry`
- `ui_entry`
- `build_entry`
- `onboarding_entry`

진입점 공통 필드:

- `id`
- `type`
- `targetEntityId`
- `projectId`
- `title`
- `reason`
- `priority`
- `sourceAdapterIds`
- `confidence`
- `metadata`

예:

- `web.xml` 기반 servlet 진입점
- `@Controller` 기반 URL 진입점
- `main` 함수 기반 실행 진입점
- 신규 입사자용 추천 시작 패키지

## 4. 설명/주석 모델

제품은 자동 분석만이 아니라 수동 보강을 포함하므로 설명 모델이 필수다.

권장 유형:

- `description`
- `note`
- `label`
- `tour_step`

공통 필드:

- `id`
- `type`
- `targetEntityId`
- `content`
- `author`
- `createdAt`
- `updatedAt`
- `source`
- `metadata`

설명:

- `source`: `auto`, `manual`, `hybrid`

이 구조가 있으면 자동 생성 설명과 사람이 수정한 설명을 함께 다룰 수 있다.

## 5. 변경 이력 모델

현재 합의에 따르면 승인 워크플로우보다 `수정 이력 관리`가 핵심이다.

따라서 메타모델에 변경 이력을 기본 포함하는 편이 좋다.

권장 필드:

- `id`
- `targetType`
- `targetId`
- `action`
- `before`
- `after`
- `author`
- `timestamp`
- `reason`
- `source`

예:

- 설명 추가
- 설명 수정
- 라벨 변경
- 온보딩 시작점 수정
- 수동 관계 보강

## 아티팩트 모델

일부 어댑터는 엔터티나 관계만이 아니라 분석 중간 산출물을 남길 필요가 있다.

예:

- 파싱된 XML bean 목록
- build target 목록
- detected route 목록

따라서 선택적으로 `artifact` 개념을 둘 수 있다.

권장 필드:

- `id`
- `type`
- `projectId`
- `producerAdapterId`
- `payload`
- `metadata`

이 아티팩트는 디버깅, 재분석, 설명 근거 표시 등에 쓸 수 있다.

## 신뢰도 모델

범용 분석기에서는 모든 결과가 같은 확실성을 가지지 않는다.

따라서 엔터티, 관계, 진입점에는 공통적으로 신뢰도 필드를 두는 편이 좋다.

권장 값:

- `high`
- `medium`
- `low`

또는 수치화:

- `0.0 - 1.0`

권장 방식은 수치값과 표시용 등급을 함께 두는 것이다.

## 원본 추적성

결과가 어디서 왔는지를 남기는 것도 중요하다.

권장 추적 필드:

- `sourceAdapterIds`
- `sourceFilePaths`
- `sourceLocations`
- `evidence`

이 구조가 있으면 사용자가 분석 결과를 신뢰하거나 검증하기 쉬워진다.

## 프로파일 특화 확장 방식

코어 메타모델은 범용이어야 하지만, 프로파일별 특수 필드는 필요하다.

예:

- Legacy Java EE:
  - servlet name
  - bean id
  - mapper namespace
  - sql statement id

- TS/JS:
  - package name
  - workspace name
  - route pattern
  - import kind

이 정보는 공통 필드를 오염시키지 않도록 `metadata` 안에 두는 편이 좋다.

## 내부 표현 예시

개념적으로는 아래 같은 구조가 적절하다.

```ts
type Entity = {
  id: string;
  type: string;
  name: string;
  displayName?: string;
  projectId: string;
  path?: string;
  language?: string;
  sourceAdapterIds: string[];
  confidence?: number;
  tags?: string[];
  metadata?: Record<string, unknown>;
};

type Edge = {
  id: string;
  type: string;
  from: string;
  to: string;
  projectId: string;
  sourceAdapterIds: string[];
  confidence?: number;
  directional?: boolean;
  metadata?: Record<string, unknown>;
};
```

## 온보딩 기능과의 연결

이 메타모델은 단순 저장 구조가 아니라 온보딩 기능의 기반이 된다.

- `entrypoint` 로 시작점 추천 가능
- `contains`, `calls`, `queries` 로 흐름 탐색 가능
- `description`, `note`, `tour_step` 으로 맥락 제공 가능
- `change_history` 로 누가 무엇을 바꿨는지 추적 가능

즉, 제품 핵심 경험 대부분이 이 메타모델 위에서 동작한다.

## 범용 분석기 관점에서의 결론

공통 메타모델은 특정 언어의 AST 자체를 그대로 저장하는 구조가 아니라, `온보딩과 구조 탐색에 필요한 공통 의미 단위`를 저장하는 구조여야 한다.

그래야 Legacy Java, TS/JS, Python 등 서로 다른 프로젝트 타입도 같은 코어 위에서 다룰 수 있다.
