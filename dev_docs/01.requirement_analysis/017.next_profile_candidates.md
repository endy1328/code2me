# Next Profile Candidates

## 목적

- `Legacy Java EE` 이후 바로 구현할 대상이 아니라, 다음 profile 경계와 입력 신호를 미리 정리한다.
- 현재 문서는 구현 시작 후의 최소 범위와 남은 확장 경계를 함께 정리한다.

## 현재 우선순위 결정

- 다음 profile 1순위: `action-family-legacy-web`
- 이유:
  - `jpetstore-6` 공개 샘플 게이트에서 현재 `legacy-java-ee`가 entry는 잡지만 action handler 복원은 못 했다.
  - Spring XML/JSP 계열보다 인접 프레임워크 경계 오탐을 먼저 줄이는 편이 현재 제품 방향에 더 직접적이다.
- `spring-java-config-web`은 2순위로 유지한다.

## 착수 기준

- 아래 3개를 만족하면 `action-family-legacy-web` 구현을 시작한다.
  - `legacy-java-ee` 합성 fixture 회귀가 안정적으로 유지된다.
  - action-family 공개 샘플의 boundary fail / no-match 패턴이 최소 3건 이상 기록된다.
  - 최소 시작 fixture 2개를 먼저 정의할 수 있다.

## 현재 판단

- 착수 상태: `started`, 현재는 `paused`
- 근거:
  - `jpetstore-6`, `spring-struts`, `form-validation`, `stripes/examples`로 경계 기록이 축적됐다.
  - `legacy-java-ee` fixture 회귀는 현재 안정 상태다.
  - 최소 시작 fixture 2개와 첫 detection / routing 복원이 이미 들어갔다.

## 최소 시작 범위

- detection 신호
  - `web.xml`의 `StrutsPrepareAndExecuteFilter`, `StripesFilter`, `DynamicMappingFilter`
  - `struts.xml`, Stripes ActionBean package 설정
- 최소 fixture
  - pure Struts action mapping fixture
  - Stripes action bean / `*.action` fixture
- 첫 완료 기준
  - 대표 entry 1개 이상 복원
  - action handler와 JSP/result 연결
  - action-family 전용 boundary에서 `legacy-java-ee`와 상호 오탐이 줄어드는지 확인

## 현재 구현 범위

- pause handoff: `dev_docs/03.development/006.action_family_pause_handoff.md`
- 새 profile id: `action-family-legacy-web`
- 새 adapter: `action-config`
- 최소 fixture
  - `samples/action-family-legacy-web-struts-minimal`
  - `samples/action-family-legacy-web-stripes-minimal`
- 현재 복원 범위
  - `web.xml` filter 기반 `*.action` entry 복원
  - `struts.xml`의 action class / method / result JSP 복원
  - Stripes `@UrlBinding`, `@DefaultHandler`, `@HandlesEvent` 기반 route / handler 복원
  - `ForwardResolution`, `RedirectResolution`, `StreamingResolution` 최소 해석
- 아직 남은 범위
  - Struts 1 계열 config
  - `DynamicMappingFilter` 계열
  - validation / form bean / forward chain 상세화
  - 공개 샘플 재검증과 boundary 보정

## Candidate A. `action-family-legacy-web`

### 목표 구조

- Struts 1, Stripes, 유사 Action 기반 레거시 웹앱

### 입력 신호 초안

- `web.xml`에 action servlet 또는 action bean/filter 선언
- `struts-config.xml`, `stripes.xml`, action bean annotation
- `.jsp`와 action URL (`*.do`, `*.action`) 조합

### 현재 공통 메타모델과의 차이

- `controller` 대신 `action` 성격이 더 강한 entry handler가 필요
- form bean / action forward / validation config를 보조 node나 metadata로 붙일 필요가 있음
- 현재 `requestHandlers` 구조는 재사용 가능하지만 mapping evidence 종류를 확장해야 함

### 현재 단계 판단

- 공개 샘플 `jpetstore-6` soft fail 결과 때문에 우선 후보로 유지

## Candidate B. `spring-java-config-web`

### 목표 구조

- XML보다 Java config가 중심인 전통 Spring MVC 웹앱

### 입력 신호 초안

- `@Configuration`, `WebMvcConfigurer`, `AbstractAnnotationConfigDispatcherServletInitializer`
- JSP view resolver bean 또는 Java config view resolver
- `@Controller`, `@RequestMapping`, JSP

### 현재 공통 메타모델과의 차이

- entry detection이 `web.xml` 중심에서 Java config bootstrap까지 확장되어야 함
- `config` node의 evidence kind에 Java config source를 추가해야 함
- request / data flow 메타모델 자체는 대부분 재사용 가능

### 현재 단계 판단

- `spring-framework-petclinic` no-match 결과 때문에 detection boundary를 넓힐 때의 2순위 후보
