# Next Profile Candidates

## 목적

- `Legacy Java EE` 이후 바로 구현할 대상이 아니라, 다음 profile 경계와 입력 신호를 미리 정리한다.
- 현재 문서는 스켈레톤만 유지하고 구현은 시작하지 않는다.

## Candidate A. `action-family-legacy-web`

### 목표 구조

- Struts 1, Stripes, 유사 Action 기반 레거시 웹앱

### 입력 신호 초안

- `web.xml`에 action servlet 또는 action bean/filter 선언
- `struts-config.xml`, `stripes.xml`, action bean annotation
- `.jsp`와 action URL (`*.do`, `*.action`) 조합

### 현재 공통 메타모델과의 차이

- `controller` 대신 `action` 성격이 더 강한 entry handler가 필요
- form bean / action forward / validation config를 보조 node나 metadata로 붙일 필요가 있음
- 현재 `requestHandlers` 구조는 재사용 가능하지만 mapping evidence 종류를 확장해야 함

### 현재 단계 판단

- 공개 샘플 `jpetstore-6` soft fail 결과 때문에 우선 후보로 유지

## Candidate B. `spring-java-config-web`

### 목표 구조

- XML보다 Java config가 중심인 전통 Spring MVC 웹앱

### 입력 신호 초안

- `@Configuration`, `WebMvcConfigurer`, `AbstractAnnotationConfigDispatcherServletInitializer`
- JSP view resolver bean 또는 Java config view resolver
- `@Controller`, `@RequestMapping`, JSP

### 현재 공통 메타모델과의 차이

- entry detection이 `web.xml` 중심에서 Java config bootstrap까지 확장되어야 함
- `config` node의 evidence kind에 Java config source를 추가해야 함
- request / data flow 메타모델 자체는 대부분 재사용 가능

### 현재 단계 판단

- `spring-framework-petclinic` no-match 결과 때문에 detection boundary를 넓힐 때의 2순위 후보
