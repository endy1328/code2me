# 분석 프로파일 공통 스키마 초안

## 문서 목적

이 문서는 code2me에서 `분석 프로파일`을 어떤 구조로 정의할지에 대한 공통 스키마 초안을 정리한 문서다.

목표는 아래를 명확히 하는 것이다.

- 프로파일이 어떤 필드를 가져야 하는가
- 프로파일이 어댑터와 어떻게 연결되는가
- 프로파일이 분석 결과 해석에 어떤 규칙을 제공하는가
- 범용 분석기 코어가 프로파일을 어떤 방식으로 실행할 수 있는가

## 프로파일의 역할 재정리

프로파일은 특정 프로젝트 타입을 어떻게 해석할지 정의하는 정책 묶음이다.

프로파일은 직접 파일을 파싱하지 않는다.

프로파일의 책임은 아래와 같다.

- 입력 소스 우선순위 정의
- 사용할 어댑터 집합 정의
- 진입점 탐색 규칙 정의
- 결과 해석 우선순위 정의
- 오탐 방지 규칙 정의
- 결과 표시 힌트 정의

즉, 프로파일은 `정책`, 어댑터는 `실행`, 메타모델은 `저장`이다.

## 프로파일 스키마 설계 원칙

- 특정 프로젝트 타입의 차이를 충분히 표현할 수 있어야 한다.
- 어댑터 조합과 실행 순서를 정의할 수 있어야 한다.
- 메타모델 해석 규칙을 연결할 수 있어야 한다.
- 자동 추천의 근거가 될 탐지 조건을 포함할 수 있어야 한다.
- UI 표시 우선순위도 프로파일 차원에서 힌트로 줄 수 있어야 한다.

## 프로파일 기본 필드

모든 프로파일은 최소한 아래 필드를 가지는 편이 좋다.

- `id`
- `name`
- `version`
- `description`
- `projectType`
- `technologyTags`
- `priority`
- `status`

예:

- `id`: `legacy-java-ee`
- `name`: `Legacy Java EE`
- `version`: `0.1.0`
- `projectType`: `enterprise-webapp`
- `technologyTags`: `java`, `ant`, `spring-mvc`, `jsp`, `mybatis`

## 탐지 관련 필드

프로파일은 사용자가 직접 고를 수도 있지만, 자동 추천도 가능해야 한다.

따라서 탐지용 조건을 가질 수 있어야 한다.

권장 필드:

- `detection.filePatterns`
- `detection.requiredSignals`
- `detection.optionalSignals`
- `detection.excludedSignals`
- `detection.scoreRules`

예:

- `**/build.xml`
- `**/web.xml`
- `**/applicationContext*.xml`
- `**/*mapper.xml`

Legacy Java EE 프로파일은 아래 같은 신호를 강하게 볼 수 있다.

- `web.xml` 존재
- Spring XML 존재
- Ant `build.xml` 존재
- JSP 존재

## 입력 소스 규칙 필드

프로파일은 어떤 입력 소스를 어떤 우선순위로 볼지 정의해야 한다.

권장 필드:

- `inputs.required`
- `inputs.optional`
- `inputs.priorityGroups`
- `inputs.exclusions`

예:

- `required`: `build.xml`, `web.xml`, `*.java`
- `optional`: `jsp`, `mapper.xml`, `properties`
- `priorityGroups`:
  - P0: 구조와 진입점
  - P1: 코드 흐름
  - P2: 데이터 접근

## 어댑터 연결 필드

프로파일은 어떤 어댑터를 필수로 요구하고 어떤 어댑터를 선택적으로 붙일지 정의해야 한다.

권장 필드:

- `adapters.required`
- `adapters.optional`
- `adapters.executionOrder`
- `adapters.parallelGroups`
- `adapters.overrides`

예:

- `required`: `ant-build-xml`, `web-xml`, `spring-xml`, `java-source-basic`
- `optional`: `jsp-view`, `ibatis-sql-map`, `mybatis-mapper`, `sitemesh-config`

## 해석 규칙 필드

프로파일의 핵심은 결과 해석 방식이다.

권장 필드:

- `interpretation.entrypointRules`
- `interpretation.relationshipRules`
- `interpretation.mergeRules`
- `interpretation.priorityRules`
- `interpretation.confidenceRules`

예:

- 진입점은 `web.xml -> spring xml -> controller` 순으로 찾는다.
- DAO 와 mapper 는 같은 persistence 흐름으로 묶는다.
- 공통 라이브러리 모듈은 비즈니스 중심축으로 우선 표시한다.

## 표시 힌트 필드

프로파일에 따라 UI에서 강조해야 하는 정보가 다르다.

권장 필드:

- `presentation.defaultViews`
- `presentation.highlightEntityTypes`
- `presentation.highlightRelationTypes`
- `presentation.recommendedEntryTypes`
- `presentation.onboardingHints`

예:

- Legacy Java EE:
  - `config`, `controller`, `service`, `dao`, `mapper`
  - 진입점과 설정 파일 강조

- TS/JS Monorepo:
  - `package`, `file`, `route`, `module`
  - workspace 경계와 엔트리 파일 강조

## 예외 처리 필드

프로파일은 자주 발생하는 오해를 방지하는 규칙도 가져야 한다.

권장 필드:

- `guardrails.doNotAssume`
- `guardrails.ignorePatterns`
- `guardrails.lowConfidencePatterns`
- `guardrails.fallbackRules`

예:

- Legacy Java EE:
  - Spring Boot 로 가정하지 말 것
  - annotation 만으로 진입점을 찾지 말 것
  - XML 설정을 우선 해석할 것

## 프로파일 내부 구조 예시

개념적으로는 아래 같은 구조가 적절하다.

```ts
type AnalysisProfile = {
  id: string;
  name: string;
  version: string;
  description: string;
  projectType: string;
  technologyTags: string[];
  detection: ProfileDetectionRules;
  inputs: ProfileInputRules;
  adapters: ProfileAdapterRules;
  interpretation: ProfileInterpretationRules;
  presentation: ProfilePresentationRules;
  guardrails: ProfileGuardrails;
  metadata?: Record<string, unknown>;
};
```

## Legacy Java EE 프로파일 예시

핵심 필드 예시는 아래처럼 잡을 수 있다.

- `id`: `legacy-java-ee`
- `technologyTags`: `java`, `ant`, `ear`, `war`, `spring-mvc`, `jsp`, `ibatis`, `mybatis`
- `inputs.required`:
  - `**/build.xml`
  - `**/web.xml`
  - `**/applicationContext*.xml`
  - `**/*.java`
- `adapters.required`:
  - `ant-build-xml`
  - `web-xml`
  - `spring-xml`
  - `java-source-basic`
- `guardrails.doNotAssume`:
  - `spring-boot`
  - `annotation-only-entrypoint`

## TS/JS Monorepo 프로파일 예시

- `id`: `ts-js-monorepo`
- `technologyTags`: `typescript`, `javascript`, `workspace`, `react`, `nextjs`, `nodejs`
- `inputs.required`:
  - `**/package.json`
  - `**/*.ts`
  - `**/*.js`
- `adapters.required`:
  - `package-json`
  - `workspace-config`
  - `ts-import-graph`

## 실행 관점에서의 의미

분석 엔진은 프로파일 스키마를 보고 아래처럼 동작하면 된다.

1. 탐지 규칙으로 후보 프로파일을 추천한다.
2. 선택된 프로파일의 입력 규칙으로 파일 우선순위를 정한다.
3. 프로파일이 요구한 어댑터를 구성한다.
4. 실행 순서와 병렬 그룹을 반영해 어댑터를 실행한다.
5. 해석 규칙으로 진입점과 중요 관계를 만든다.
6. 표시 힌트에 맞춰 결과를 정리한다.

즉, 프로파일 스키마가 있어야 범용 코어가 프로젝트 타입별 동작을 통제할 수 있다.

## 최소 스키마와 확장 스키마

초기에는 아래 정도만 있어도 운영 가능하다.

- `id`
- `name`
- `detection`
- `inputs`
- `adapters`
- `interpretation`

이후 확장 가능한 항목:

- `presentation`
- `guardrails`
- `qualityChecks`
- `incrementalRules`
- `profileCompatibility`

## 범용 분석기 관점에서 중요한 이유

프로파일 스키마가 명확해야 아래가 가능해진다.

- 새 프로젝트 타입 빠른 추가
- AI 기반 프로파일 생성 자동화
- 프로파일 간 비교와 테스트
- 추천 시스템 구축
- 어댑터 재사용 극대화

즉, 프로파일 스키마는 범용 분석기의 확장성을 결정하는 핵심 인터페이스다.

## 현재 우선 적용 방향

현재는 `Legacy Java EE 프로파일`을 먼저 상세화하는 것이 맞지만, 스키마 자체는 다른 프로젝트 타입도 바로 수용할 수 있도록 범용적으로 잡아야 한다.
